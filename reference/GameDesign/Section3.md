# 22. The Game Design Thinking Process

## 核心流程

> **Find the Game → Define the Pillars → Lock → Ideate → Define → Prototype**

前两步确定游戏基础，确认后原则上锁定；后三步围绕具体功能反复迭代。

## 1. Find the Game

明确：

* 这是什么游戏？
* 面向哪些玩家？
* 核心吸引力是什么？
* 想提供什么体验？

**产物：Concept Document / 10-pager**

只需描述高层概念，不要过早深入具体参数和功能细节。

## 2. Define the Pillars

定义游戏不可轻易妥协的核心原则、体验目标和约束，通常为 **3～5条**。

好的支柱应当：

* 含义具体，有明确说明
* 能指导功能取舍
* 彼此不冲突
* 能防止设计偏离与范围膨胀

**产物：Pillars Document**

支柱过多会限制创意并容易冲突；过少或过于模糊则容易导致 Scope Creep。

## 3. Lock

概念与支柱确认后，不应因为某个功能失败而频繁修改。

只有核心体验本身被证明存在重大问题时，才考虑重新设计。

## 4. Ideate

在概念和支柱的约束下，为具体功能提出多个候选方案。

**产物：候选设计方案**

## 5. Define

将选定方案写成可实施的规格，包括：

* 规则与参数
* 玩家操作与反馈
* 潜在体验问题
* 所需资产
* 技术实现问题
* 与其他系统的关系

**产物：Feature Spec / GDD章节**

只需写到足以沟通和实现，避免在验证前过度细化。

## 6. Prototype

实现可测试版本，对比：

* 文档中写的内容
* 设计者的预期
* 实际游玩体验

发现问题后返回 Ideate 或 Define，继续迭代。

## 检查表

* 能否简洁说明游戏概念和核心体验？
* 是否有3～5条清晰且不冲突的支柱？
* 每个新功能支持哪条支柱？
* 文档是否足以让团队实施？
* 是否在验证前投入了过多细节？
* 原型具体要验证什么假设？
* 实际体验是否符合设计预期？

# 23. Game Concept: Maslow’s Hierarchy of Needs

## 核心概念

马斯洛需求层次可用于思考游戏的：

* 主题与氛围
* 玩家目标
* 成长过程
* 资源和系统

这些层级并非严格按顺序出现，也可以同时存在。

| 层级                 | 核心需求    | 常见游戏内容        |
| ------------------ | ------- | ------------- |
| Physiological      | 生存      | 食物、水、住所、健康    |
| Safety             | 安全与稳定   | 基地、防御、收入、秩序   |
| Social             | 归属与关系   | 队伍、社区、角色关系    |
| Esteem             | 成就与认可   | 排名、称号、声望、高难挑战 |
| Self-actualisation | 创造与自我实现 | 探索、建造、创作、自我表达 |

## 设计应用

游戏可以围绕某一层需求展开，也可以形成逐层发展的体验：

> 生存 → 稳定 → 建立关系 → 获得认可 → 自由创造

同一系统也可以满足多个层级。例如玩家创作并分享关卡，同时涉及：

* 自我实现：表达创意
* 社交：与他人分享
* 尊重：获得评价与认可

## 检查表

* 游戏主要围绕哪一种需求？
* 玩家开局缺少什么？
* 玩家长期追求什么？
* 玩家目标是否会随进度变化？
* 机制是否真正支持主题？
* 游戏是否提供成长、探索、创造或新的体验视角？

# 24. Game Concept: The Beginner’s Mindset

## 核心原则

不要默认玩家拥有和设计者相同的：

* 游戏经验
* 类型知识
* 操作习惯
* 专业知识
* 兴趣和偏好

设计时应从目标玩家，尤其是新玩家的视角重新检查游戏。

## 初学者视角三问

在游戏任意节点暂停并询问：

1. 玩家是怎样到达这里的？
2. 玩家需要知道什么，才能到达这里？
3. 玩家从哪里获得了这些知识？

知识来源可能包括：

* 游戏内教程
* UI与反馈
* 试错
* 同类游戏经验
* 现实常识
* 外部攻略

关键问题是：

> 游戏真的教过玩家，还是设计者只是认为玩家应该知道？

## 玩家知识分类

### 可以合理假设已知

目标玩家普遍熟悉的惯例，只需轻量提示。

### 部分玩家可能知道

需要通过提示、操作和反馈进行确认。

### 游戏独有知识

必须明确教学，并提供实际练习机会。

## Extreme Users

研究同类游戏中极度正面和极度负面的评价。

* 正面评价：发现玩家最珍视的体验和类型惯例
* 负面评价：发现痛点、误解、UI和教学问题

应寻找反复出现的模式，而不是根据单条评论修改设计。

## 检查表

* 目标玩家是谁？
* 当前节点要求玩家掌握哪些知识？
* 每项知识在哪里被传达？
* 玩家是否有机会实际练习？
* 是否存在未解释的术语、图标或规则？
* 是否过度依赖同类型游戏经验？
* 是否因为自己过于熟悉而低估了学习难度？
* 同类游戏评价中有哪些重复出现的优点和痛点？

# 25. Game Pillar Document

## 核心概念

Game Pillar Document 不需要复杂。

重点是让所有成员明确：

* 每条支柱具体是什么意思
* 为什么它重要
* 它如何影响设计和开发决策

不能只写一个模糊标题，每条支柱都应配有简短说明。

# 26. Ideate: Ideation Techniques

## 核心原则

Ideation 是在明确约束下产生大量候选方案。

所有想法都应服务于：

* 游戏概念
* 游戏支柱
* 目标玩家
* 当前需要解决的问题

> Ideate负责产生选项，Define负责筛选和完善。

## Brainstorming规则

### 开始前

* 明确要解决的问题
* 提供游戏概念、支柱、类型和限制
* 邀请会受到该功能影响的相关成员
* 设置2～3个限时区块，每段15～45分钟

### 进行中

* 不打断发言
* 先听完整想法
* 暂时禁止“做不到”“不可能”等否定
* 使用“Yes, and...”继续发展他人的想法
* 鼓励古怪但仍与问题相关的方案
* 完整记录
* 将产生想法和评价想法分开

## Brainwriting

参与者先独立、安静地写下想法，再统一展示和讨论。

优点：

* 减少强势成员主导
* 让不擅长口头表达的人参与
* 适合远程协作
* 通常能提高想法数量和多样性

基本流程：

> 独立写想法 → 集中展示 → 合并分类 → 补充组合 → 之后筛选

## SCAMPER

适合改进已有机制或系统。

| 字母 | 方法                      | 核心问题            |
| -- | ----------------------- | --------------- |
| S  | Substitute 替换           | 可以用什么替换？        |
| C  | Combine 组合              | 可以和什么结合？        |
| A  | Adapt 借用／适配             | 可以借鉴什么并进行调整？    |
| M  | Modify 修改               | 可以放大、缩小或改变什么？   |
| P  | Put to another use 改作他用 | 还能用来做什么？        |
| E  | Eliminate 删除            | 去掉它会怎样？         |
| R  | Reverse 反转              | 顺序、关系或规则反过来会怎样？ |

## 筛选检查表

* 是否解决了原始问题？
* 是否支持游戏概念和支柱？
* 是否适合目标玩家？
* 是否与其他系统冲突？
* 是否存在更简单的方案？
* 实现成本是否合理？
* 是否值得进入 Define 和 Prototype？
* 保留它是因为有效，还是仅仅因为“听起来很酷”？

## 简化执行流程

1. 写出一个具体问题
2. 列出相关支柱与限制
3. 限时产生多个方案，不立即评价
4. 必要时使用 Brainwriting 或 SCAMPER
5. 合并、分类并筛选
6. 只选择少量方案进入 Define


# 29. Define: The 5 Phases of Players’ Decision-Making

## 核心流程

一次完整的玩家决策包含五个阶段：

> **决策前信息 → 告知可用 → 执行动作 → 产生后果 → 传达后果**

## 1. Before：决策前

玩家需要获得足以判断的信息。

* 信息过多：造成认知超载
* 信息过少：变成盲目决策
* 目标：信息足够，但不过量

## 2. Availability：告知决策可用

游戏需要让玩家知道现在可以做什么。

常见方式：

* UI按钮或图标
* 视觉提示
* 音效
* 状态变化
* 已建立的游戏惯例

必须清楚区分“可执行”和“不可执行”。

## 3. Action：执行动作

玩家通过按键、点击或菜单作出选择。

设计重点：

* 减少操作步骤
* 避免界面杂乱
* 输入速度应匹配游戏节奏
* 不让操作本身妨碍思考

## 4. Consequences：产生后果

选择必须改变游戏状态，例如：

* 数值或资源变化
* 角色关系变化
* 单位、物品或事件出现
* 路线或世界状态改变

没有实际后果的选择不具备真正的意义。

## 5. Feedback：传达后果

玩家必须知道自己的选择造成了什么。

可通过：

* UI变化
* 动画
* 音效或语音
* 文字提示
* 进度条
* 世界与角色反应

## 检查表

* 玩家是否掌握足够信息？
* 玩家是否知道当前有哪些选择？
* 操作是否直接、清楚？
* 选择是否真正改变游戏状态？
* 玩家是否能理解发生了什么？
* 玩家是否知道结果与自己的选择有关？

# 30. Define: Automatic vs. Rational Decision-Making

## 两种决策系统

### Automatic System：自动系统

* 快速、直觉、接近反射
* 消耗较少注意力
* 依赖习惯、经验和模式识别
* 常用于熟练操作和高压决策

### Rational System：理性系统

* 较慢，会分析和比较
* 消耗更多认知资源
* 用于学习新规则、制定策略和权衡后果
* 容易因为疲劳或复杂度过高而把决定交给自动系统

## Bounded Rationality：有限理性

玩家无法掌握所有信息，只能在有限的：

* 信息
* 时间
* 注意力
* 经验

之下，选择一个“看起来足够合理”的方案。

不完整信息应制造可判断的不确定性，而不是让玩家纯粹猜测。

## 学习与熟练化

常见的玩家学习过程：

> **理性学习 → 反复练习 → 自动执行 → 引入新挑战**

玩家成长不仅是数值提高，也包括操作和判断逐渐自动化。

## 设计原则

* 快节奏游戏更依赖自动系统
* 策略和回合制游戏更依赖理性系统
* 底层操作可以自动化
* 高层战术与战略应保留思考空间
* 决策时间必须匹配复杂度

## 检查表

* 希望玩家直觉反应，还是认真分析？
* 决策时间是否足够？
* 高压环境中是否出现过多新信息？
* 新能力是否提供了学习和练习过程？
* 玩家掌握旧内容后，是否出现新的挑战？
* 不确定性是可推理的，还是只能猜？
* 失败后玩家能否理解原因？

# 31. Define: Interactive Loops for Decision-Making

## 基本结构

互动循环是：

> **玩家形成假设 → 采取行动 → 系统反馈 → 玩家修正理解**

游戏通过不同时间尺度的循环，持续让玩家学习、判断和行动。

## 互动循环类型

| 循环                | 时间尺度    | 主要内容           |
| ----------------- | ------- | -------------- |
| Action / Feedback | 不到1秒    | 攻击、闪避、瞄准等即时反应  |
| Short-term        | 数秒至数分钟  | 当前战术、资源使用、短期目标 |
| Long-term         | 一局至多个周目 | 战略、成长路线、长期规划   |
| Emotional         | 不固定     | 价值观、同情、内疚、身份认同 |
| Social            | 游戏内外    | 社会制度、群体和现实比较   |
| Cultural          | 游戏之外    | 创作者、时代、文化与社会影响 |

## 设计原则

### Action / Feedback

* 反馈必须快速、明确
* 长时间连续使用会造成疲劳
* 应与较平缓的阶段交替

### Short-term

* 提供当前挑战所需的信息
* 让玩家进行低负担的战术判断

### Long-term

* 当前选择应影响未来
* 结果可以延迟，但需要阶段性反馈
* 玩家应能根据新情况调整计划

### Emotional / Social / Cultural

* 情感重量需要角色、叙事、美术和氛围共同铺垫
* 不能只靠文字声称某个选择“很艰难”
* 同一个选择可以同时存在于多个循环

## 检查表

* 当前机制主要属于哪种循环？
* 玩家多久收到一次反馈？
* 反馈是否能帮助玩家修正理解？
* 即时操作是否持续过久？
* 是否有短期战术和长期目标？
* 长期选择是否有阶段性反馈？
* 是否存在超越纯收益的情感选择？
* 情感后果是否有足够铺垫？

# 32. Define: Bad Decision-Making

## 1. Blind Decisions：盲目决策

玩家缺乏判断所需的信息，只能猜测。

常见原因：

* 选择出现得太早
* 选项含义不清楚
* 后果无法预估
* 玩家尚未理解相关系统

改善方法：

* 延后重要选择
* 先让玩家体验相关机制
* 提供基本信息与预期后果
* 允许之后调整或修正

## 2. Obvious Decisions：显而易见的决策

存在一个全面优于其他选项的答案。

例如两个选项只有数值高低差异，没有任何代价或适用场景区别。

改善方法：

* 加入真实取舍
* 设计不同使用场景
* 加入成本、风险或机会成本
* 避免长期存在唯一最优策略

## 3. Meaningless Decisions：无意义决策

### 没有实际后果

选择没有改变：

* 系统状态
* 资源
* 关系
* 路线
* 难度
* 过程或演出

### 有后果，但没有反馈

系统发生了变化，但玩家无法感知或建立因果关系。

## Agency：玩家能动性

玩家应感觉：

> 我的决定会影响游戏的过程、状态或结果。

主线最终汇合并非一定有问题，但不同选择至少应改变中间过程或局部体验。

## 检查表

* 玩家是否有足够依据判断？
* 选择是否出现得过早？
* 是否存在全面优于其他选项的答案？
* 不同选项是否有不同用途？
* 选择是否真正改变系统？
* 玩家是否能感知变化？
* 后果是否太晚，导致因果关系不清？
* 选择是否让玩家感受到能动性？

# 33. Define: Interesting Decisions

## 1. Trade-offs：取舍

一个选择同时具有优势和劣势。

作用：

* 支持不同玩法风格
* 避免唯一正确答案
* 增加构筑和重复游玩价值

设计重点：

* 优点和缺点都必须真正影响玩法
* 缺点不能只是无关痛痒的装饰

## 2. Risk–Reward：风险与回报

玩家在高风险高收益与低风险低收益之间选择。

设计重点：

* 风险和奖励需要匹配
* 玩家应大致理解自己承担的风险
* 风险不可理解会变成盲选
* 风险过低、收益过高会变成明显答案

## 3. Yomi：预测对手

玩家通过观察对手的信息，预测其行动并调整策略。

成立条件：

* 可以获得部分情报
* 获得情报后可以调整方案
* 不存在固定的万能策略
* 对手既有规律，也不能完全预测

## 4. Planning：规划

玩家根据当前信息，为长期目标安排：

* 资源
* 路线
* 成长
* 行动顺序

规划成立的前提：

* 玩家理解基础规则
* 长期目标明确
* 当前行动与未来结果存在联系
* 进度可被观察
* 计划可以调整

## 对照表

| 方法          | 核心问题           |
| ----------- | -------------- |
| Trade-offs  | 我愿意付出什么代价？     |
| Risk–Reward | 我愿意承担多少风险？     |
| Yomi        | 对方准备做什么？       |
| Planning    | 现在怎样做才能实现未来目标？ |

## 检查表

* 每个选项是否都有真实优缺点？
* 是否支持不同玩法或目标？
* 风险与收益是否匹配？
* 玩家是否理解风险？
* 玩家能否获取并利用对手信息？
* 当前选择是否影响未来？
* 玩家能否看到规划进展？
* 计划失败后是否仍有调整空间？

# 34. Define: Self-Determination Theory

## 核心概念

玩家的内在动机主要来自：

> **Autonomy 自主感 + Competence 胜任感 + Relatedness 归属感**

## 1. Autonomy：自主感

玩家感觉自己能够控制行为、目标和游玩方式。

支持自主感：

* 有实际后果的选择
* 多种可行玩法
* 可自行安排目标和成长顺序
* 明确目标下存在不同解决方式

削弱自主感：

* 过度手把手引导
* 选择没有影响
* 只有一种被认可的正确玩法
* 强迫玩家按固定顺序行动

自主感不等于完全没有限制。规则和目标仍需清楚。

## 2. Competence：胜任感

玩家通过学习和练习感到自己越来越熟练。

理想循环：

> **理解 → 练习 → 掌握 → 新挑战**

设计重点：

* 难度应与玩家能力匹配
* 失败应提供学习信息
* 掌握旧内容后再引入变化
* 胜任感不仅来自数值成长，也来自玩家自身能力提升

## 3. Relatedness：归属感

玩家感觉自己与：

* 其他玩家
* 游戏角色
* 阵营或群体
* 游戏世界
* 游戏社区

存在真实联系。

可通过：

* 合作与互助
* 公会、队伍和共同目标
* 资源赠送或师徒机制
* 可信的角色关系
* 游戏外社区和创作分享

联网本身并不会自动产生归属感。

## 内在动机与外在奖励

金币、经验、签到奖励只能提供外在刺激，不能代替：

* 我的选择有意义
* 我真的掌握了系统
* 我在乎这些角色或群体

## 检查表

### 自主感

* 玩家是否拥有真正有效的选择？
* 是否允许不同玩法？
* 是否过度限制行动顺序？
* 目标明确的同时，解决方式是否自由？

### 胜任感

* 玩家能否逐渐理解和掌握机制？
* 挑战是否与能力匹配？
* 是否能感受到自身进步？
* 失败是否能帮助学习？

### 归属感

* 玩家是否与角色、群体或世界建立联系？
* 社交功能是否鼓励实际合作？
* 群体是否拥有共同身份与目标？
* 单机角色关系是否可信？
* 是否存在交流、分享或创作空间？
