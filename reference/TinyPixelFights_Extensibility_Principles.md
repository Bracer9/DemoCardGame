# Tiny Pixel Fights — 机制延展性原则

> 目的：把后续增加技能、状态、天气、遗物、Cost/AP 变化时的设计思路固定下来。  
> 本文不是立即重构计划，而是讨论和实装新机制前的共同判断基准。

## 1. 为什么现在要明文化

Tiny Pixel Fights 已经从简单的卡牌攻击游戏，变成一个有状态、盾、预见、反击、绝对伤害、在线同步、动画和音效的战斗系统。之后如果继续加入天气、遗物、更多技能、可变 Cost、全局规则变化，就不能再用“某个技能里写一个 if”解决。

好的延展性不是把项目做成大型卡牌引擎，而是保证每个机制都有清楚的归属：

- 基础数据归基础数据。
- 临时状态归状态系统。
- 全局规则归战场或遗物系统。
- 行动是否合法由统一计算结果决定。
- UI 和预测显示同一套规则结果。

如果规则分散在攻击按钮、预测接口、实际攻击结算、前端显示和日志里，每次改 Cost 或 AP 都会牵一发动全身。

## 2. 核心原则

### 2.1 默认通用，例外显式

系统属性应当描述通用概念，而不是描述某个技能的目标。

好的例子：

- `IsDispellable`：这个状态是否可被驱散。
- `IsBuff`：这个状态是否是正面状态。
- `DamageType`：伤害类型。

不好的例子：

- `IsDruidTarget`
- `IsAttackBuffDispelTarget`
- `CanBeRemovedByWeakeningSpores`

原因是后者把状态系统绑死在某个技能上。未来如果加入沉默、净化、偷取 Buff、强化 Buff 效果，又会继续长出一堆技能专用属性。

当前确认原则：

- Buff 默认可驱散。
- 不可驱散 Buff 需要显式声明，例如 `beast-rage`。
- 德鲁伊当前只处理 `IsBuff && IsDispellable` 的目标。

### 2.2 基础值不是最终值

角色卡上的攻击、生命、Cost 是基础值，不应该被当成最终值。

最终数值应当可以由多个来源叠加：

```text
最终 Cost
= 角色基础 Cost
+ 角色自身状态修正
+ 队伍光环修正
+ 战场 / 天气修正
+ 遗物 / 装备 / 事件修正
+ 当前行动类型修正
+ 临时一次性修正
```

攻击、防御、反击、伤害、AP 上限也应遵循类似思想。

这不代表现在立刻实装完整 modifier 系统，而是之后任何会改变 Cost/AP 的机制，都应该朝这个方向放，而不是在各处硬改。

### 2.3 规则来源要分层

未来可能出现的规则来源可以分为：

| 来源 | 例子 | 生命周期 | 影响范围 |
|---|---|---|---|
| 角色基础定义 | 怪兽 Cost 3，攻击 3 | 永久 | 单角色 |
| 角色状态 | 野兽之怒、衰弱、春播 | 按状态规则 | 单角色或来源指定 |
| 队伍光环 | 公主祝福、占卜师预见 | 来源在场时 | 己方全体 |
| 战场 / 天气 | 本回合全员 Cost -1 | 回合或阶段 | 双方或指定阵营 |
| 遗物 / 装备 | 全员 Cost +1 但攻击翻倍 | 战斗内或长期 | 队伍 / 全局 |
| 公共指令 | 防御阵型 | 当前回合或直到消耗 | 队伍 |
| 临时结算效果 | 下一次攻击 Cost -1 | 一次性 | 行动级 |

设计新机制时，应先判断它属于哪一层。不要把天气写成角色 Buff，也不要把角色专属状态写成全局规则。

## 3. Cost/AP 可变性的概念模型

用户提出的例子：

- 怪兽获得野兽之怒后，AP 消耗 +1。
- 天气系统：全员本回合 Cost -1。
- 遗物系统：全员 Cost +1，但攻击力翻倍。

这些都属于 Cost/AP 经济系统的不同来源。合理设计时，不应直接修改 `CharacterDefinition.Cost`，因为那是基础卡面值。

更合理的概念是：

```text
CharacterDefinition.Cost = 卡面基础 Cost
EffectiveCost = 当前上下文中计算出的实际 Cost
```

实际行动时应统一使用 `EffectiveCost`：

- 是否可行动：`EffectiveCost <= current AP`
- 预测面板：显示本次行动实际消耗
- 攻击执行：扣除实际 Cost
- UI 卡面：基础 Cost 和当前 Cost 如果不同，应能提示差异
- 在线同步：双方看到同一个计算结果

### 3.1 Cost Modifier 应该具备的信息

未来如果实装，不一定要一开始做复杂接口，但概念上每个 Cost 修正至少需要这些信息：

| 字段 | 用途 |
|---|---|
| 来源 ID | 例如 `beast-rage`、`weather-tailwind`、`relic-blood-oath` |
| 数值 | `+1`、`-1`、`set to 1` 等 |
| 适用对象 | 自身、己方、敌方、全员、某类型角色 |
| 适用行动 | 主动攻击、防御指令、技能、全部行动 |
| 生命周期 | 持续、回合结束、一回合一次、触发后消失 |
| 显示文本 key | UI/hover/预测能解释原因 |

### 3.2 修正顺序要可解释

Cost 和 AP 不宜随意叠加到负数或 0。需要提前约定下限和顺序。

建议初步规则：

```text
1. 从基础 Cost 开始
2. 加总 additive modifier，例如 +1 / -1
3. 再处理 multiplier 或特殊 set 效果
4. 最终 Cost 至少为 1，除非某机制明确写成免费行动
```

如果以后有“免费攻击一次”，它应该是明确的特殊机制，而不是 Cost 被各种 -1 意外压到 0。

## 4. 攻击力、伤害和 Cost 不要混在一起

“全员 Cost +1 但攻击力翻倍”这种遗物同时影响两个维度：

- 行动经济：Cost 变高，行动次数减少。
- 伤害输出：攻击力翻倍，单次行动更强。

这应该被表示成两个独立 modifier，而不是一个巨大 if：

```text
Relic A:
- CostModifier: all allies +1
- AttackModifier: all allies ×2
```

这样未来 UI 也能分别解释：

- “此角色因遗物 Cost +1”
- “此角色因遗物攻击翻倍”

玩家才会觉得规则清楚，而不是数值凭空改变。

## 5. UI 和预测必须吃同一套计算

任何 Cost/AP 变化都必须同时影响：

1. 卡面显示的当前 Cost。
2. 是否能点击/拖动/攻击。
3. 预测面板的 AP 消耗。
4. 后端实际扣除 AP。
5. 在线对手看到的状态。
6. hover 详情里列出的原因。

如果预测用一套逻辑，实际攻击用另一套逻辑，就会出现“看起来能动，点了不能动”或“预测消耗 2，实际扣 3”的灾难。

因此未来更好的方向是后端提供结构化结果，例如：

```text
effectiveCost: 4
baseCost: 3
costModifiers:
  - source: beast-rage
    amount: +1
    labelKey: status.beast-rage
```

前端只负责展示，不重新推导规则。

## 6. 当前项目中的可类比经验

### 6.1 IsDispellable 的教训

这次德鲁伊与野兽之怒的问题说明：

- “攻击强化 Buff 是否能被德鲁伊清除”不应该是德鲁伊专用属性。
- 正确抽象是“这个 Buff 是否可驱散”。
- 德鲁伊只是使用这个通用属性的一个技能。

以后遇到类似问题，应先问：

```text
这个属性是在描述系统本身，
还是只是在描述某个技能想怎么处理它？
```

如果答案是后者，就要重新考虑抽象。

### 6.2 共享盾的教训

共享盾和有恃无恐曾经混在一起，导致同一个效果被扣两次。后来拆成：

- 共享盾：队伍层资源，吸收伤害。
- 有恃无恐：角色身上的一次性反击惩罚状态。

这说明：当一个机制包含两个功能时，最好拆成两个可见、可追踪的状态或资源。

## 7. 新机制进入项目时的检查表

每次加入新技能、天气、遗物、事件、Cost/AP 改动前，先填这张表：

| 问题 | 回答 |
|---|---|
| 机制来源是什么？角色、状态、天气、遗物、公共指令，还是战斗规则？ |
| 生命周期是什么？永久、回合内、一次性、来源离场失效？ |
| 影响对象是谁？自身、己方、敌方、全员、某类型？ |
| 影响哪个数值？Cost、AP 上限、攻击、反击、伤害、治疗、盾？ |
| 是否需要参与预测？ |
| 是否需要在 hover 详情中显示？ |
| 是否需要事件 icon / 音效 / 动画？ |
| 是否会制造固定最优解？ |
| 是否会拖慢对局？ |
| 是否需要新增通用属性，还是已有属性足够？ |
| 是否会影响在线同步？ |

## 8. 对 Codex 的长期要求

之后讨论或实装任何新机制时，Codex 应默认采取以下步骤：

1. 先复述机制意图，而不是马上写代码。
2. 判断它属于哪个规则层。
3. 检查是否已有通用属性或管线可复用。
4. 如果需要新属性，优先命名为系统概念，而不是技能概念。
5. 明确预测、实际结算、UI、动画、音效、文档、测试是否都受影响。
6. 先给最小实现路径，不要一上来重构成大型框架。

一句话总结：

> 新机制可以复杂，但它进入系统的接口必须清楚、通用、可解释。
