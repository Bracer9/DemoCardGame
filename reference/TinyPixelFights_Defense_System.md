# Tiny Pixel Fights — 物理防御 / 魔法防御实装文档

> 日期：2026-06-28  
> 阶段：PvP Growth Prototype Phase 1 — Defense Lab  
> 目的：把物理 / 魔法伤害分类从“图标信息”推进到“目标选择、职业身份和未来成长系统可使用的基础参数”。

## 1. 第一版目标

这不是完整装备、防御穿透或天气系统。第一版只做一个小而稳定的规则地基：

- 每个角色拥有 `PhysicalDefense` 和 `MagicalDefense` 两个基础参数。
- 伤害结算时，按照伤害类型扣减对应防御。
- 防御必须进入攻击预测、实际结算、卡面 UI、hover 详情和数值表。
- 绝对伤害不受防御影响。
- 数值极保守，避免当前 1～5 攻击区间被固定减伤彻底压死。

## 2. 规则定义

### 2.1 基础公式

```text
类型防御后伤害 = max(0, 原始伤害 - 目标对应类型防御)
```

- 物理伤害读取目标 `PhysicalDefense`。
- 魔法伤害读取目标 `MagicalDefense`。
- 绝对伤害不读取防御。
- 如果未来状态、遗物或战场效果让有效防御变成负数，则视为对应类型易伤：只有原本大于 0 的伤害会增加 `abs(负防御)`。

### 2.2 结算顺序

第一版采用以下顺序：

```text
攻击 / 技能产生基础伤害
→ 攻击方输出修正
→ 目标类型防御
→ 预见 / 守护等技能修正
→ 共有盾吸收
→ 盾后技能修正
→ 扣 HP
```

设计理由：

- 类型防御是角色自身的基础耐性，应早于共有盾。
- 共有盾是队伍资源，应吸收“已经被目标自身耐性处理后的伤害”。
- 守护是否触发仍以类型防御和共享盾之后是否还有有效物理主攻伤害为准，避免无意义触发。

### 2.3 与现有机制的关系

| 机制 | 第一版关系 |
|---|---|
| 预见 | 在类型防御后继续判定 -1。预测显示仍保留概率范围。 |
| 共有盾 | 在类型防御与部分技能减伤后吸收剩余伤害。 |
| 骑士守护 | 只对主动物理攻击，且类型防御/盾后仍有伤害时触发。骑士肩代わり承受的 1 点物理伤害不是绝对伤害，会继续受到骑士自身物防等防御管线影响。 |
| 反击 | 反击也是物理或魔法伤害，因此同样吃对应防御。 |
| 德鲁伊衰弱 | 先降低攻击输出，再由目标防御扣减。 |
| 怪物绝对伤害 | 不受物防、魔防、预见、盾、守护影响。 |

## 3. 第一版初始数值

| 角色 | 物防 | 魔防 | 设计意图 |
|---|---:|---:|---|
| 農民 | 0 | 0 | 普通单位，不增加额外复杂度。 |
| 姫 | 0 | 0 | 低基础体量，用技能补偿，不靠防御。 |
| 魔法使い | 0 | 0 | 高攻击角色保持脆弱，不先给魔防。 |
| 占い師 | 0 | 1 | 魔法相性核心，拥有轻微魔法耐性。 |
| 騎士 | 1 | 0 | 重装守护者，物理耐性明确。 |
| ドルイド | 0 | 1 | 自然 / 魔法侧辅助，轻微魔法耐性。 |
| バーバリアン | 0 | 0 | 高攻进攻者，不给稳定防御。 |
| モンスター | 1 | 0 | 大型肉体威胁，轻微物理耐性。 |

暂不使用 2 点防御。原因：当前攻击区间很低，2 点固定减伤会让低攻角色更容易失去行动价值。

## 4. UI 要求

### 4.1 卡面

卡面需要显示物防 / 魔防：

- 放在卡片左下角附近。
- 字体风格与左上角攻击力保持一致。
- 用 `P` / `M` 或小型标签区分物防、魔防。
- 只显示数字，不使用现代 UI 框。
- 如果有效防御为负数，卡面显示红色数字，不显示负号。

### 4.2 Hover / Selected Inspector

左侧角色详情中加入：

```text
P.DEF  物防值
M.DEF  魔防值
```

后续如果防御可被 buff、装备、天气修改，应显示当前值与基础值差异，例如 `1 (+1)`。如果当前值为负数，显示为红色绝对值，表示易伤而不是普通防御。

### 4.3 攻击预测

预测面板需要反映防御后的伤害范围，并在脚注中说明：

- 目标防御降低了主动伤害。
- 攻击方防御降低了受到的反击。

第一版可先用简单文本说明，不必做专属图标演出。

## 5. 架构原则

第一版虽然只做基础属性，但命名要为未来 modifier 留路：

- `CharacterDefinition.PhysicalDefense`
- `CharacterDefinition.MagicalDefense`
- `GameEngine.GetPhysicalDefense(CharacterState)`
- `GameEngine.GetMagicalDefense(CharacterState)`
- `GameEngine.GetDefense(CharacterState, DamageType)`

未来若加入状态、装备、天气、遗物、防御穿透，不应直接改 `CharacterDefinition`，而应在 `GetDefense` 管线中叠加来源。

当前 `GetDefense` 已允许返回负数，实际伤害结算由统一管线决定：正数防御减伤，负数防御增加对应类型伤害。新增来源时优先加入 getter / modifier 管线，不要在攻击、预测或 UI 中分别硬写。

## 6. 验收标准

- 后端实际攻击结算会按伤害类型扣减防御。
- 攻击预测与实际伤害一致。
- 卡面能看到物防 / 魔防。
- hover 详情能看到物防 / 魔防。
- 绝对伤害不受防御影响。
- `dotnet build` 通过。
- `node --check wwwroot/app.js` 通过。

## 7. Playtest 观察点

- 1 攻角色是否被物防 / 魔防过度压制？
- 骑士、怪物的物防是否让物理队打不动？
- 占い師、ドルイド的魔防是否让魔法内战更有相性？
- 对局时长是否明显增加？
- 玩家是否能通过预测和卡面解释“为什么少了 1 点伤害”？
