# Tiny Pixel Fights — 共有盾防御层改造草案

> 日期：2026-06-29  
> 状态：需求确认 / 实装前设计  
> 目的：把当前“玩家共享盾值”升级为“拥有自身防御参数的队伍防御层”，同时修正盾与物防/魔防的结算直觉。盾不作为普通攻击可选实体。

---

## 1. 为什么要改

当前共有盾在视觉上是展开在队伍前方的蓝色屏障，但实际规则是：

```text
伤害 → 目标物防/魔防 → 共有盾 → HP
```

这在数值上合理，但和玩家直觉不完全一致。既然盾画在队伍前方，它应当先承受外部攻击。角色防御应该表现为“攻击穿透盾之后，角色自身再减伤”。

此外，未来系统会需要：

- 角色技能或行动效果直接强化、驱散或破坏盾。
- 盾拥有自己的物理防御、魔法防御。
- 某些攻击/技能可以绕过盾。

因此不应继续把盾只当作 `PlayerState.SharedShield` 一个整数；但也暂时不需要把盾做成角色式实体或普通攻击目标。

---

## 2. 新规则口径

### 2.1 普通伤害结算顺序

新顺序：

```text
原始伤害
→ 共有盾先吸收
→ 若仍有剩余，目标物防/魔防减伤或增伤
→ 若仍有主动物理伤害，骑士守护等最后防线生效
→ 扣 HP
```

简化理解：

- 盾先挡外部攻击。
- 角色防御挡穿透伤害。
- 骑士是最后一层保护。

### 2.2 盾自身防御

共有盾拥有自己的：

- `PhysicalDefense`
- `MagicalDefense`

第一版默认均为 `0`。

盾受到伤害时先使用盾自身对应防御处理打到盾上的伤害：

```text
原始伤害
→ 盾物防/魔防
→ 盾值吸收剩余伤害
→ 打穿后，剩余伤害进入目标角色物防/魔防
```

例：

```text
敌人对骑士造成 4 物理伤害
我方盾值 2，盾物防 1，骑士物防 1

4 物理
→ 盾物防 1，剩 3
→ 盾值吸收 2，剩 1
→ 骑士物防 1，剩 0
→ 骑士不掉血
```

## 3. 盾作为队伍防御层

### 3.1 后端概念

建议新增或抽象出类似：

```csharp
public sealed class TeamShieldState
{
    public int Value { get; set; }
    public int PhysicalDefense { get; set; }
    public int MagicalDefense { get; set; }
    public bool IsActive => Value > 0;
}
```

挂在 `PlayerState`：

```csharp
public TeamShieldState Shield { get; set; }
```

短期可以保留 `SharedShield` 作为兼容字段，但新增逻辑应逐步转向 `PlayerState.Shield.Value`。如果改动范围可控，也可以一次性迁移。

### 3.2 目标模型

普通攻击仍然选择角色，不选择盾。  
未来如果有“强化盾”“破盾”“驱散盾”等角色技能，可以通过角色 hover / 技能区域 / 行动按钮触发，并把效果直接作用于目标玩家的 `TeamShieldState`。

如果未来确实需要更通用的选择模型，可以再引入 `TargetRef`，但第一版不把盾纳入普通目标选择系统。

---

## 4. 攻击与盾的关系

### 4.1 选择敌方角色时

当敌方共有盾存在时：

- 玩家仍然选择敌方具体角色作为攻击目标。
- 只要攻击没有“无视盾”属性，伤害仍会先打在目标所属队伍的盾上。
- 攻击光线继续连接到被选择角色，预测窗口负责明确显示“先击中共有盾”。

这保证视觉和规则一致：

```text
我选中了敌方法师
→ 攻击线连到法师
→ 伤害先撞到敌方盾
→ 打穿后才进入法师自身防御与 HP
```

### 4.2 对盾直接生效的技能/行动

如果未来角色拥有“破盾”“强化盾”“给盾加物防/魔防”等能力：

- 不通过普通攻击选盾。
- 在该角色的 hover / 详情 / 技能操作区域中点击对应能力。
- 效果直接指定目标玩家的盾状态。
- 这类效果可以绕开普通攻击预测，也可以弹出独立确认窗口，视技能复杂度决定。

---

## 5. 伤害包需要预留的通用 flag

为了避免之后为每个技能写特例，建议 `DamagePacket` 或攻击请求预留：

```csharp
public bool IgnoresShield { get; init; }
public bool AffectsShieldOnly { get; init; }
```

含义：

| Flag | 用途 |
|---|---|
| `IgnoresShield` | 伤害绕过盾，直接进入角色防御/HP。适合刺杀、真实命中、部分魔法。 |
| `AffectsShieldOnly` | 效果只作用于盾，不会打穿后继续伤害角色。适合破盾技、削盾技。 |

第一版不一定全部做完整 UI，但后端结构应预留这些概念。

---

## 6. 与现有技能的关系

### 骑士守护

新顺序下：

```text
伤害 → 盾 → 目标物防/魔防 → 若仍有主动物理伤害 → 骑士守护
```

骑士肩代わり的 1 点物理伤害不是绝对伤害，应继续走骑士自身物防/魔防管线。也就是说骑士物防足够时，可能不掉血。

### 怪兽「美女与野兽」

当前机制是：对非公主主动攻击造成 0 伤害时，追击绝对伤害。

新盾规则下需要明确：

- 如果攻击被盾完全挡住，且目标是非公主角色，主动攻击对目标造成 0 伤害，仍应满足触发条件。
- 追击绝对伤害默认是否无视盾？建议保持当前“绝对伤害直接扣目标 HP，不受盾、防御、buff/debuff 影响”。
- 如果未来某个破盾技能只对盾生效，不应触发美女与野兽，因为它不是对角色的主动攻击。

### 德鲁伊、魔法师、狂战士等

- 以“对目标角色造成伤害”为条件的技能，应区分“对盾造成伤害”和“对角色造成 HP 伤害”。
- 以“攻击命中/攻击无效”为条件的技能，需要在预测与日志中写清楚。
- 这要求日志事件最好能区分：
  - shield damage
  - character damage
  - shield broken
  - carried-over damage

## 7. UI / UX 需求

### 7.1 盾 hover HUD

当共有盾存在时，hover 盾弧或防御相关区域，应出现和防御阵型同 style 的 HUD。

显示内容：

- 当前盾值。
- 盾物防。
- 盾魔防。
- 当前层数 / 是否强化。
- 简短规则说明：盾先吸收，打穿后才进入角色防御。

### 7.2 攻击预测

预测窗口必须显示：

```text
盾吸收：X
盾后剩余：Y
目标防御减少：Z
最终 HP 伤害：N
```

如果攻击无视盾，应明确显示“无视共有盾”。

### 7.3 攻击光线

当敌方有盾：

- 攻击光线仍连接到被选择角色。
- 预测中明确显示“先击中盾”。
- 如果未来某技能无视盾，光线可以使用不同颜色/穿透效果。

## 8. 实装建议顺序

### Step 1：后端数据结构

- 新增 `TeamShieldState`。
- 让 `PlayerState` 持有盾对象。
- 保持 DTO 暴露当前盾值、防御值。

### Step 2：伤害结算管线

- 把共有盾结算移动到目标角色防御之前。
- 盾结算时使用盾自身物防/魔防。
- 打穿盾后，再对目标角色应用物防/魔防。
- 骑士守护保持在最后。

### Step 3：预测同步

- `AttackPreviewService` 使用同一套结算顺序。
- 预测显示盾吸收、防御削减、最终伤害。

### Step 4：前端选择和 HUD

- 盾作为可 hover 的战场防御层，但不是普通攻击目标。
- 盾 HUD 显示盾值与防御。
- 普通攻击仍选择角色，预测负责说明盾结算。

### Step 5：文档与表格

- 更新 `TinyPixelFights_Shield.md`。
- 更新 `TinyPixelFights_Defense_System.md`。
- 更新 `TinyPixelFights_Skills.md` 中怪兽、骑士相关描述。
- 如改变数值权重，更新 balance 表。

---

## 9. 第一版暂不做的事

为了避免范围爆炸，第一版不做：

- 给盾做复杂技能树。
- 盾拥有独立 HP / 最大 HP / 死亡状态。
- 盾作为完整角色进入队伍 slot。
- 盾作为普通攻击可选目标。
- 多层盾独立分层计算。
- 盾被治疗、复活、异常状态等完整角色化逻辑。

第一版只把盾作为“队伍级防御层”。

---

## 10. 设计原则总结

新共有盾应遵守：

```text
盾是队伍前方的障碍物。
角色防御是盾被穿透后的个人耐性。
骑士守护是最后一层保护。
特殊技能通过通用 flag 表示是否无视盾、只影响盾或强化盾。
```

这样既符合视觉直觉，也为未来“盾防御强化”“破盾技能”“无视盾技能”等机制留下空间，同时避免把盾过早复杂化成完整实体。
