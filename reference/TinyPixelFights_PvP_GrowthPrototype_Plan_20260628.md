# Tiny Pixel Fights — PvP Growth Prototype Plan

> 日期：2026-06-28  
> 状态：阶段计划 / 接入顺序  
> 目标：把“英雄 + 普通兵 + 局内成长 + 奖励选择”的方向逐步接入当前对战模式，同时避免一次性塞入过多系统。  
> 相关文档：`GDD.md`、`reference/TinyPixelFights_GDD_Brainwriting_20260627.md`、`reference/TinyPixelFights_External_Inspiration_Brainwriting_20260627.md`、`reference/TinyPixelFights_Extensibility_Principles.md`

---

## 0. 本计划的核心判断

当前项目长期方向以单机为主，但现在的工作重心仍是对战模式。对战模式的职责不是完整复制未来爬塔流程，而是验证未来单机战斗会依赖的核心体验：

> 玩家是否愿意在一场短局战斗中，通过奖励、成长和队伍构成，把一支小队塑造成有方向的战术单位？

因此，系统接入顺序不按“最终游戏最酷的样子”排序，而按下面的原则排序：

1. 先接入依赖少、风险低、能单独验证的机制。
2. 先验证核心乐趣，再扩展内容量。
3. 先让玩家看懂，再追求深度。
4. 每次只让一个新系统承担主要风险。
5. 保留 Classic 4 Hero 作为对照基线，新增系统尽量进入实验模式或开发开关。

## 1. 当前暂定路线一句话

```text
Classic 4 Hero 基线
-> 物理/魔法防御小实验
-> 英雄/普通兵分类
-> 1 英雄 + 普通兵开局
-> 固定 round 三选一奖励
-> 轻量 BP 作为成长货币
-> 英雄第一次升级二选一
-> 普通兵军衔
-> 小型遗物
-> 兵种技能与副官
-> 英雄转职
-> 天气/战场规则
```

其中，**指挥等级系统暂缓**。第一阶段不需要重型等级条；如果固定奖励窗口 + 轻量 BP 已经能产生局内成长快感，就没有必要过早引入复杂经济系统。

## 3. Phase 0：基线保护与实验入口

### 目标

在引入新机制前，保留当前对战作为可比较的基线。

### 范围

- 保留当前 Classic 4 Hero 对战。
- 新机制尽量通过实验模式、配置开关或独立队伍生成逻辑接入。
- 每次 playtest 能回答：“新机制比 Classic 多了什么？少了什么？”

### 成功信号

- 新机制失败时可以回到旧模式继续测试。
- 平衡问题能被定位到具体实验层，而不是所有机制混在一起。
- 用户和开发者能清楚知道当前玩的是什么规则集。

### 暂不做

- 不做完整模式选择 UI。
- 不做玩家可配置的高级规则大厅。
- 不做最终产品形态的模式包装。

## 4. Phase 1：物理防御 / 魔法防御小实验

### 目标

验证物理 / 魔法伤害分类是否能从“图标信息”变成“目标选择和职业身份”。

### 第一版范围

- 新增物理防御、魔法防御概念。
- 数值极保守：多数单位为 0，少量单位为 1，极少数测试 2。
- 防御只服务于预测和伤害结算，不同时接入遗物、天气、副官。
- 先用现有 8 英雄进行 Defense Lab。

### 推荐测试口径

```text
骑士：物防 1
占卜师：魔防 1
魔法使：物防 0 / 魔防 1 或 0
公主、农民、德鲁伊等低攻单位：谨慎观察是否被防御废掉
```

具体数值必须拉表，不在 GDD 中写死。

### 成功信号

- 玩家能解释为什么某次伤害降低。
- 玩家会考虑用物理打低物防目标、用魔法打高物防目标。
- 低攻击角色仍然有行动价值。
- TTK 没有明显拖长。

### 风险

- 固定减伤对当前 1～4 攻击区间影响很大。
- 与共有盾、预见、守护的结算顺序会变复杂。
- 如果预测面板不解释清楚，玩家只会觉得伤害“不对”。

### 暂不做

- 不做防御穿透。
- 不做装备提供防御。
- 不做天气修改防御。
- 不做大量防御型遗物。

---

## 5. Phase 2：英雄 / 普通兵数据分类

### 目标

先建立系统语言：哪些单位是英雄，哪些单位是普通兵。不要等到成长系统加入时再临时补标签。

### 第一版范围

- 现有 8 名角色长期视为 `hero`。
- 新增普通兵时标记为 `soldier`。
- 保留伤害类型标签：`physical`、`magical`、特殊情况为 `absolute` 或规则内特殊伤害。
- 为未来奖励池、遗物、副官、成长条件准备稳定 ID 和标签。

### 成功信号

- 奖励池可以按 `hero` / `soldier` / `physical` / `magical` 过滤。
- UI 可以用清楚图标或短标签区分英雄和普通兵。
- 后端规则不需要用角色名硬编码“谁能升级、谁能成为副官”。

### 暂不做

- 不做阵营大系统。
- 不做复杂职业树。
- 不做羁绊系统。

---

## 6. Phase 3：1 英雄 + 普通兵开局

### 目标

验证“少量英雄 + 简单普通兵”是否能降低开局认知负担，同时让英雄更特殊。

### 第一版范围

每方队伍改为实验配置：

```text
1 英雄 + 3 普通兵
```

普通兵先做 4 类即可：

| 普通兵 | 定位 | 设计目的 |
|---|---|---|
| 剑兵 | 标准物理攻击 | 提供简单基准 |
| 盾兵 | 低攻高 HP / 物防 | 测试防御参数和承伤 |
| 术士 | 魔法攻击 | 激活物理/魔法目标选择 |
| 牧师 | 小治疗或驱散 | 避免治疗只属于公主 |

### 成功信号

- 新玩家更快理解场面。
- 英雄行动更有仪式感。
- 普通兵不只是无聊血包。
- 队伍里有不同目标优先级。

### 关键观察

- 1 英雄是否太少，导致早期看点不足。
- 普通兵是否过于同质。
- 治疗普通兵是否削弱公主身份。
- 低技能普通兵是否让战斗变得过素。

### 暂不做

- 不做普通兵军衔。
- 不做兵种技能。
- 不做副官。
- 不做普通兵随机招募。

---

## 7. Phase 4：固定 round 三选一奖励框架

### 目标

先验证“局内奖励选择”本身是否好玩，再决定 BP、指挥等级或复杂经济是否必要。

### 第一版范围

```text
Round 2 / 4 / 6 后触发奖励选择。
每名玩家看到 3 个奖励。
每次奖励选择拥有 1 次刷新机会。
选择后立即生效。
```

如果实际对局较短，可以先只做 Round 2 / 4。

### 奖励类别

第一版奖励池只需要少量、清楚、可解释：

| 类别 | 示例 | 目的 |
|---|---|---|
| 即时战术 | 立即获得护盾、治疗一个单位 | 救局和短期选择 |
| 小属性成长 | 某单位 HP +2 或攻击 +1 | 测试成长反馈 |
| 英雄成长候选 | 英雄第一次升级入口 | 连接 JRPG 快感 |
| 普通兵成长候选 | 普通兵 Rank +1 | 给士兵未来价值 |
| 小遗物候选 | 每回合首次物理攻击 +1 | 初步测试 build |

### 随机性原则

- 奖励可随机，但必须三选一。
- 每次至少保证不同类别或不同对象，避免三个选项都像同一个。
- 英雄升级这类关键奖励需要隐藏保底或权重保护。
- 不向玩家展示复杂概率表，但 UI 要解释每个奖励的实际效果。

### 成功信号

- 玩家看到奖励时有期待。
- 三个选项之间存在真实取舍。
- 刷新机会有决策价值。
- 奖励选择不严重拖慢回合节奏。

### 暂不做

- 不做庞大奖励池。
- 不做高稀有度史诗遗物。
- 不做高级 Class。
- 不做奖励之间的复杂连锁。

## 8. Phase 5：轻量 BP 作为成长货币

### 目标

验证 BP 是否能给奖励选择增加资源取舍，而不把战斗变成刷分游戏。

### 当前定位

BP 第一版不是表现评分，不是指挥等级经验，也不是复杂战术目标系统。

BP 第一版应定义为：

> 有保底的局内成长货币。

### 推荐第一版规则

```text
回合或 round 补给：+1 BP。
战斗推进：本回合首次造成生命伤害或击破敌方共有盾，+1 BP。
主动奖励每回合最多一次。
```

奖励选择仍由固定 round 触发；BP 只决定玩家买得起哪些奖励，或是否愿意刷新 / 保存资源。

### 推荐奖励消费口径

```text
普通小奖励：低 BP 或免费。
普通兵成长：低 BP。
英雄第一次升级：中 BP。
稀有遗物：高 BP。
```

### 成功信号

- 玩家能用一句话理解 BP 来源。
- 主动进攻有一点节奏优势。
- 玩家不会为了刷 BP 做明显不合理行动。
- 落后方不会因为拿不到 BP 被排除在成长系统之外。

### 风险

- 如果主动 BP 太高，会奖励领先者继续领先。
- 如果低保 BP 太高，BP 会退化成直接发奖励。
- 如果奖励价格太复杂，玩家会把注意力从战斗转到算账。

### 暂不做

- 不做指挥等级。
- 不做每种行动都给 BP。
- 不做击杀大量给 BP。
- 不做公开复杂战术目标列表。
- 不做累计 BP 和可用 BP 双轨，除非后续证明必要。

## 9. Phase 6：英雄第一次升级二选一

### 目标

验证“英雄从白卡到觉醒技能”的局内成长快感。

### 第一版范围

- 只做英雄第一次升级。
- 升级后从两个基础技能中选择一个。
- 先选 2～4 名英雄做完整闭环，不需要 8 名英雄同时完成。
- 当前已有技能可以作为其中一个默认方向，另一个方向用于测试新玩法。

### 推荐第一批英雄

| 英雄 | 理由 |
|---|---|
| 公主 | 治疗和指挥方向差异明显 |
| 农民 | 资源/成长主题清楚 |
| 魔法使 | 魔法伤害与状态好测试 |
| 骑士 | 防御、守护和物防系统相关 |

### 成功信号

- 玩家记得自己选择了哪个技能。
- 同一英雄两个技能路线会影响后续行动。
- 对手能看懂英雄已经觉醒了什么。
- 升级不是单纯数值 +1，而是角色身份变化。

### 暂不做

- 不做第二次属性升级。
- 不做第三次转职。
- 不做新立绘。
- 不做全英雄完整技能树。

---

## 10. Phase 7：普通兵军衔

### 目标

验证普通兵是否能成为玩家愿意培养和保护的对象。

### 第一版范围

```text
Rank 0：基础普通兵。
Rank 1：获得一次小属性成长。
Rank 2：暂不一定解锁技能，可先作为满级状态。
```

第一版可以只做数值成长：

- 攻击 +1
- HP +2 或 +3
- 对应防御 +1
- 治疗量 +1

### 成功信号

- 玩家会有意识地把奖励投给某个普通兵。
- 普通兵成长后能改变目标优先级。
- 普通兵没有抢走英雄的主角感。

### 暂不做

- 不做副官。
- 不做兵种技能。
- 不做复杂个人贡献进度。

---

## 11. Phase 8：小型遗物

### 目标

验证全局 modifier 是否能改变队伍 build，而不破坏预测和可读性。

### 第一版范围

- 每方最多 1～2 个遗物。
- 遗物池不超过 6 个。
- 全部遗物效果必须朴素、可预测、可 hover。

### 推荐第一批

| 遗物 | 效果方向 |
|---|---|
| 老兵磨刀石 | 每回合首次物理主动攻击 +1 |
| 星银护符 | 每回合首次受到魔法伤害 -1 |
| 备用绷带 | 回合开始治疗最低 HP 友方 1 |
| 民兵号角 | 普通兵成长时获得小额 BP 或额外 HP |
| 破盾楔 | 每回合首次击破共有盾后，下一次 HP 伤害 +1 |
| 战术旗布 | 英雄行动后，下一个普通兵主动攻击 +1 |

### 成功信号

- 玩家能围绕遗物调整行动顺序。
- 遗物改变策略，但不直接判定胜负。
- 预测面板能列出遗物修正。

### 暂不做

- 不做史诗遗物。
- 不做负面代价遗物。
- 不做遗物槽替换 UI。
- 不做遗物与天气联动。

---

## 12. Phase 9：兵种技能与副官

### 目标

验证“培养普通兵，再把普通兵转化为英雄强化”是否有策略和情感价值。

### 第一版副官范围

- 只有 Rank 满级普通兵可以成为副官。
- 成为副官后，该普通兵从战场单位转为附着在英雄上的强化来源。
- 第一版副官只提供简单属性或弱化版兵种能力。

### 推荐第一版副官效果

| 副官 | 英雄获得 |
|---|---|
| 剑兵 | 主动攻击 +1，每回合最多一次 |
| 盾兵 | 物防 +1 或首次物理伤害 -1 |
| 术士 | 魔法主动攻击 +1 |
| 牧师 | 回合开始回复 1，或行动后治疗相邻友方 1 |

### 成功信号

- 玩家愿意保护某个普通兵等其成为副官。
- 副官化有清楚机会成本：少一个场上单位，换英雄长期强化。
- 英雄因此形成不同 build。

### 暂不做

- 不做英雄专属副官协同。
- 不做副官牺牲保命。
- 不做多个副官槽。
- 不做副官长期继承到单机 run 外。

---

## 13. Phase 10：英雄第二次升级与高级 Class

### 目标

在英雄第一次升级被证明有趣后，再验证 JRPG 转职是否值得内容成本。

### 推荐接入方式

先只选 1～2 名英雄做完整三段成长：

```text
第一次：基础技能二选一。
第二次：属性成长。
第三次：高级 Class 二选一，换立绘或职业纹章。
```

### 成功信号

- 玩家会期待某个英雄进入高级 Class。
- 高级 Class 改变玩法，而不只是更大数字。
- 对手能提前看到转职威胁，并有窗口应对。

### 暂不做

- 不做 8 名英雄同时转职。
- 不做每个高级 Class 全套新动画。
- 不做高成本新立绘批量接入，除非规则先证明好玩。

---

## 14. Phase 11：天气 / 战场规则

### 目标

验证每局变化是否能增加计划感，而不是制造不可读混乱。

### 接入前置条件

天气必须等以下系统稳定：

- 预测面板能显示多来源 modifier。
- 奖励和成长系统已有基本可读性。
- 玩家已能理解普通兵、英雄成长、遗物的基本层级。

### 第一版范围

- 只做 3～4 种天气。
- 显示当前天气和下一天气。
- 每局最多出现 2～3 次天气变化。
- 天气效果最多一两条规则。

### 推荐天气

| 天气 | 效果方向 |
|---|---|
| 顺风号令 | 双方 AP 上限 +1 |
| 烟尘 | 反击伤害 -1 |
| 暴雨 | 物理主动攻击 -1 |
| 魔潮 | 魔法主动攻击 +1 |

### 暂不做

- 不做随机突然天气。
- 不做天气与 BP、遗物、副官全部联动。
- 不做地形 slot 系统。

## 15. 横向实现检查表

每个新机制进入实现前，都需要检查这些面：

| 面向 | 必须回答 |
|---|---|
| 后端规则 | 状态如何存储？结算顺序在哪里？是否复用通用 modifier？ |
| 预测 | 攻击预测能否显示该机制的影响？ |
| UI | 卡面、hover、奖励框、日志是否能解释？ |
| 本地化 | 文本是否进入 `wwwroot/locales/*.json`？ |
| 在线同步 | 双方是否看到同一状态和奖励结果？ |
| 日志 / 回放 | 关键变化是否有事件记录？ |
| 动画 / 音效 | 玩家是否能感知发生了什么？ |
| 测试 | 是否有单元测试或至少可复现的手动测试步骤？ |
| 文档 | GDD 或 reference 是否需要同步更新？ |

---

## 16. 最大风险与控制策略

| 风险 | 控制策略 |
|---|---|
| 系统过多，玩家不知重点 | 每阶段只引入一个主系统，其他系统保持静态 |
| BP 滚雪球 | BP 以低保为主，主动奖励每回合上限，避免击杀大量给 BP |
| 奖励随机导致挫败 | 三选一 + 一次刷新 + 隐藏保底 |
| 英雄升级抽不到 | 英雄升级加入权重保护，不让玩家完全失去核心成长路径 |
| 普通兵无聊 | 普通兵先用清楚定位区分，不急着加复杂技能 |
| 普通兵抢主角感 | 英雄承担技能、故事、转职；普通兵承担结构、承伤、辅助和副官材料 |
| 防御拖局 | 数值上限保守，先测 TTK |
| 遗物/天气破坏可读性 | 晚接入，必须进入预测和 hover |

## 17. 阶段决策门

每个阶段结束后，至少回答四个问题：

1. 这个机制是否支持 GDD 的设计支柱？
2. 玩家是否能在不读 rulebook 的情况下理解它？
3. 它是否创造了取舍，而不是固定最优解？
4. 它是否值得进入下一阶段，还是应该缩小、替换或删除？

如果答案不清楚，优先继续小规模 playtest，而不是马上扩展内容量。

## 18. 近期最小可执行路线

最推荐的近期路线是：

```text
1. 更新 GDD，确认对战成长原型方向。
2. 新增本 Plan 文档，固定接入顺序。
3. 做 Defense Lab：物防/魔防小数值实验。
4. 建立 hero / soldier 标签和普通兵数据。
5. 做 1 英雄 + 3 普通兵实验队伍。
6. 做固定 round 三选一奖励框架，先用少量简单奖励。
7. 接入轻量 BP，作为奖励消费货币。
8. 做 2～4 名英雄的第一次升级二选一。
```

这条路线的目标不是把最终游戏做完，而是尽快验证一个小闭环：

> 一名英雄带着普通兵进入战斗，通过短局奖励获得成长，并让下一轮战斗选择发生变化。

如果这个闭环成立，再考虑副官、遗物、转职和天气。

## 19. 当前不建议立刻做的事

- 不立刻做完整指挥等级。
- 不立刻做 8 英雄 × 2 高级 Class。
- 不立刻做大遗物池。
- 不立刻做天气和地形。
- 不立刻做副官牺牲、羁绊、长期伤痕等情感系统。
- 不把 BP 做成复杂战术得分表。
- 不让奖励系统替代战斗本身。

## 20. 计划总结

下一阶段最重要的不是“加更多机制”，而是把机制压成可验证的顺序：

```text
先让队伍结构变化。
再让奖励选择出现。
再让轻量 BP 增加取舍。
再让英雄第一次成长。
最后才扩展普通兵、遗物、副官、转职和天气。
```

这能最大程度保护当前对战原型，同时为未来单机爬塔和 JRPG 成长方向打下真实、可 playtest 的战斗基础。
