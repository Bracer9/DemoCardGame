# Tiny Pixel Fights PvP Growth Prototype Plan

Last updated: 2026-06-28

## 0. 当前结论

这份计划已经按新的依赖关系重排：**BP 系统必须先做**。原因很简单，后面的三选一奖励、招募普通兵、获取新英雄、英雄升级素材、普通兵强化素材、副官、遗物和天气测试，都需要一个能稳定显示、积累、扣除、被日志解释的局内资源作为底座。

物防/魔防已经作为战斗基础规则进入项目，不再把它放在成长系统路线的主干第一位。接下来真正的 growth prototype 起点是 BP。

本计划服务于当前对战 prototype，而不是一次性做完未来单机爬塔。对战模式的职责是验证：局内成长是否能让战斗更有长期规划、更多逆转点，同时不把玩家和开发都拖进不可控复杂度。

## 1. 当前设计支柱

1. **局内成长要可感知**：玩家在一局对战中要明显感到自己的队伍变强、选择变多，而不是只是在数值上偷偷变化。
2. **BP 是成长选择的燃料**：BP 不再绑定指挥等级，也不承担复杂等级条功能；它直接用于三选一奖励的购买、跳过和攒资源决策。
3. **对战 prototype 先验证机制是否有趣**：先用 dummy 奖励确认 BP 的显示、获取和扣除都稳定，再接真实英雄/普通兵成长。
4. **先做通用管线，再塞内容**：英雄升级、普通兵叠加强化、新英雄获取、遗物，都应该接同一套奖励窗口、费用、稀有度和日志解释。
5. **每个 phase 都必须能玩**：每一步结束都应该有一个可以启动、可以打一局、可以观察问题的版本，而不是半截系统。

## 2. 被放弃或降级的方向

曾讨论过“指挥等级”作为局内成长进度，但当前判断是开发量、UI 解释、数值规划复杂度都过高，不进入当前 prototype。后续如果需要表达“局内阶段推进”，优先使用固定 Round 奖励窗口、BP 成本、奖励池稀有度和保底规则，而不是再引入独立等级系统。

## 3. BP Core 规则口径

初始规则先保持保守，目标是“让 BP 有动力，但不让刷 BP 成为唯一最优解”。

- 初始 BP：5
- BP 上限：10
- 每名玩家独立拥有 BP
- 每个玩家自己的回合开始获得低保 +1 BP
- 单回合 BP 获取上限：3
- 破坏敌方共有盾：+1 BP
- 主动攻击造成生命伤害：+1 BP
- 发动角色主动技能：+2 BP
- 角色被动技能实际触发：+1 BP
- 所有来源都受单回合获取上限 3 限制
- BP 消耗不会影响已经发生过的成长，只影响当前是否买得起奖励

实现时要注意：技能 BP 不应该因为“按钮可点”而获得，必须是技能实际发动并进入战斗日志；被动技能也必须有明确事件，避免一个被动在同一连锁里重复刷分。

## 4. 分阶段总览

| Phase | 名称 | 主要目标 | 完成后玩家能测试什么 |
| --- | --- | --- | --- |
| 0 | Baseline Lock | 确认当前物防/魔防、AP、盾、伤害预测、日志可作为基线 | 现有战斗规则没有被成长系统破坏 |
| 1 | BP Core | 接入 BP 状态、UI、获取、上限、单回合上限、日志 | 玩家能打一局确认 BP 正常增长 |
| 2 | Dummy Reward Window | 固定 Round 的三卡奖励窗口，只做费用、扣 BP、reset、skip | 玩家能确认奖励节奏和扣费体验 |
| 3 | Card Taxonomy & Skill Lock | 区分英雄/普通兵，建立技能锁定/解锁和多技能槽数据结构 | 未升级英雄技能被锁，普通兵更简单 |
| 4 | Start Draft | 开局随机 3 英雄选 1，再从 4 个初始兵种选 2 | 玩家能用新开局流程进入战斗 |
| 5 | Real Reward Integration | BP 奖励接入新英雄、英雄升级、普通兵获取、普通兵强化 | 玩家能用 BP 真的改变队伍 |
| 6 | Adjutant | 满足条件的普通兵可在己方回合附到英雄上 | 玩家能测试副官强化和 UI 表达 |

| 8 | Weather / Battlefield | 天气或战场规则进入固定 Round 或奖励池实验 | 玩家能测试临时全局规则变化 |

## 5. Phase 0 - Baseline Lock

目标不是新增内容，而是把当前战斗基线稳定住，避免 BP 后续接入时不知道问题来自哪里。

需要确认：

- 物理/魔法/绝对伤害分类清晰。
- 物防/魔防已经进入实际伤害和预测。
- 共有盾、破盾、反击、预见、衰弱、治疗不会因为 BP 事件监听而改变原本结果。
- 现有日志能支持新增 BP 事件。
- 当前 UI 仍然能容纳一个新的玩家资源显示。

完成标准：

- `dotnet build` 通过。
- `node --check wwwroot/app.js` 通过。
- 至少手动打一局，确认没有明显回归。

## 6. Phase 1 - BP Core

这是接下来第一个开发 phase。

### 6.1 数据与规则

新增玩家级 BP 状态：

- `currentBp`
- `maxBp = 10`
- `bpGainedThisTurn`
- `bpGainCapPerTurn = 3`
- 每个玩家独立记录

BP 获取应通过统一方法结算，例如 `TryGainBp(playerId, amount, reason)`。不要在攻击、技能、被动里各自硬改 BP，否则后面很难做日志、动画和上限。

### 6.2 获取时机

第一版先接这些事件：

- 己方回合开始：低保 +1
- 敌方共有盾被破坏：+1
- 主动攻击造成生命伤害：+1
- 主动技能实际发动：+2
- 被动技能实际触发：+1

如果一个行动同时满足多个条件，可以逐项尝试获得 BP，但最终受 `bpGainedThisTurn <= 3` 限制。例如一次主动技能既造成伤害又破盾，理论来源可能超过 3，但本回合最终最多只进账 3。

### 6.3 UI

BP 必须在主战斗画面常驻显示，不能藏在日志里。建议：

- 玩家自己的 BP 放在右下区域，靠近 YOU / 玩家信息，但避开手牌扇形区域。
- 敌方 BP 放在右上区域，靠近 ENEMY / 玩家信息。
- 使用圆形金币/徽章感的“BP 令牌”，风格接近旧金、黑铁、羊皮纸，不做现代进度条。
- 数字显示当前 BP，例如 `BP 5`。
- hover 显示本回合已获得多少、上限多少，以及最近一次 BP 来源。

### 6.4 日志与反馈

至少需要能在日志里解释：

- 回合开始获得 BP
- 破盾获得 BP
- 造成生命伤害获得 BP
- 主动技能获得 BP
- 被动触发获得 BP
- 因本回合上限未获得完整 BP
- 因 BP 上限未获得完整 BP

第一版动画可以很轻，只要数字变化清楚即可；之后再加飞向 BP 徽章的粒子或小光点。

### 6.5 Phase 1 手动测试

测试重点：

- Round 开始后当前玩家 BP +1。
- BP 不超过 10。
- 单回合最多只获得 3。
- 多来源叠加时不会突破上限。
- 对手回合获得的是对手 BP，不会串号。
- 主动技能和被动技能只有实际发动才给 BP。
- UI 与日志一致。

完成标准：

- 能完整打一局。
- BP 能稳定积累。
- 没有奖励系统也不影响现有胜负流程。

## 7. Phase 2 - Dummy Reward Window

目标是先测试“奖励窗口作为系统流程”是否成立，不急着接真实内容。

### 7.1 触发节奏

第一版按正式规则接：

- Round 1 出现第一次奖励选择。
- 之后每隔 2 Round 出现一次奖励选择。
- 双方各自独立选择。

如果实现中 Round 1 首窗影响开局节奏，可以保留配置开关，方便临时改成 Round 2 开始测试；但文档默认规则仍是 Round 1 首窗。

### 7.2 Dummy 奖励

奖励先是 dummy card，不改变战斗，只验证流程：

- Dummy Reward A：消耗 2 BP
- Dummy Reward B：消耗 4 BP
- Dummy Reward C：消耗 6 BP

点击后只扣 BP、写日志、关闭窗口。买不起的奖励显示为不可购买，但仍可看到费用。

### 7.3 Reset / Skip

每次奖励窗口：

- 玩家有 1 次 reset 机会。
- reset 后重新随机 3 张 dummy 奖励。
- skip 不消耗 BP，也不获得奖励。
- skip 的意义是让玩家攒 BP，避免被迫买低价值奖励。

完成标准：

- 奖励窗口能按 Round 节奏出现。
- BP 扣除正确。
- reset 只能用一次。
- skip 保留 BP。
- 双方选择互不干扰。

## 8. Phase 3 - Card Taxonomy & Skill Lock

这一步只做结构，不急着填满所有英雄分支。

### 8.1 卡牌类型

引入最小分类：

- Hero Card：英雄卡，拥有故事、技能成长、升级路径。
- Soldier Card：普通兵卡，属性直观，技能少或没有技能。

普通兵第一版按四类准备：

- 物理高攻脆皮
- 物理高防弱攻
- 魔法高攻脆皮/高魔防
- 辅助/牧师/术士

### 8.2 英雄技能锁定

英雄要支持：

- 技能槽上限扩展到 2 个。
- 初始英雄的第一个技能默认锁定。
- 英雄卡面能显示锁定技能，但不允许发动。
- 第一次英雄升级时，在两个初级技能中选择一个解锁。
- 第三次英雄升级时，在两个终极方向中选择一个。

注意：这一步可以先只做数据结构和 UI 锁，不需要所有英雄都填满两个初级技能和两个终极技能。

完成标准：

- 英雄/普通兵能被规则层区分。
- 锁定技能不会出现在可用行动里。
- hover 能解释“未解锁”。
- 现有英雄的当前技能可以迁移为未来的初级技能候选之一。

## 9. Phase 4 - Start Draft

目标是替换“一开始直接发 4 张英雄”的高认知负担开局。

### 9.1 开局选择

开局流程：

1. 每名玩家从随机 3 个英雄中选择 1 个。
2. 每名玩家从 4 个初始兵种中选择 2 个。
3. 进入战斗时，队伍包含 1 英雄 + 2 普通兵。

是否补到 4 个单位，可以先不做；如果战斗系统强依赖 4 格，可以临时给一个默认普通兵或空位规则，但要在文档里标注为过渡实现。

### 9.2 测试重点

- 随机 3 英雄不会重复。
- 双方各自选择互不干扰。
- 英雄未升级时技能保持锁定。
- 普通兵技能少、信息少，确实降低开局认知负担。
- 开局后现有战斗循环仍能正常运行。

## 10. Phase 5 - Real Reward Integration

这一 phase 才把 BP 和真实成长连起来。不要在 BP Core 阶段提前塞真实内容，否则很难定位 bug。

### 10.1 奖励类型

第一批真实奖励可以包括：

- 获取新的普通兵卡。
- 获取新的英雄卡。
- 英雄升级素材。
- 普通兵升级/强化素材。
- 少量通用资源或临时补偿奖励。

### 10.2 英雄升级

英雄最多 3 次升级：

- 第一次：选择并解锁一个初级技能。
- 第二次：基础属性提升。
- 第三次：选择终极方向/高级 class，并获得对应高级技能或机制。

实现时建议先做“升级素材/升级机会”的通用奖励，再决定奖励是否直接指定某个英雄。这样后续可以做保底、稀有度和定向选择。

### 10.3 新英雄获取

获取新英雄的意义是扩大队伍构筑，而不是让手牌爆炸。

需要限制：

- 队伍/上场单位上限仍以 4 为基准。
- 新英雄可以进入后备、手牌或候选区，具体表现要和当前代码结构匹配。
- 如果上限已满，玩家需要选择替换、保留在后备，或暂时不能领取。

### 10.4 普通兵获取与强化

普通兵重复获得时提供两种玩家选择：

- 叠加到已有同名普通兵上，作为强化/升阶素材。
- 不叠加，作为独立单位保留。

这样玩家可以在“少数精锐”和“多单位横向展开”之间做选择。上限仍然要保护，不能无限铺单位。

完成标准：

- BP 能购买真实奖励。
- 英雄升级能正确改变技能锁定和属性。
- 新英雄能进入队伍系统。
- 普通兵能获取、叠加、或作为独立单位存在。
- 奖励日志能解释玩家到底买到了什么。

## 11. Phase 6 - Adjutant

副官系统放在普通兵成长之后，因为它依赖“普通兵已经有等级/军衔/满级状态”。

第一版规则：

- 满足条件的普通兵可成为副官。
- 玩家在自己回合通过手牌/队伍操作把普通兵附到英雄上。
- 副官离开独立单位位置，成为英雄的附属强化。
- 副官提供基础属性或一个简单被动增益。

第一版副官增益建议非常保守：

- 物理兵：英雄物理攻击 +1 或物理防御 +1。
- 魔法兵：英雄魔法攻击 +1 或魔防 +1。
- 辅助兵：英雄获得一次性治疗增强或回合开始少量回复。
- 防御兵：英雄获得受击减伤或护盾交互。

完成标准：

- 副官附加和解除规则明确。
- UI 能显示英雄已有副官。
- 属性变化进入预测。
- 不能通过反复附加/解除刷 BP 或刷触发。

## 12. Phase 7 - Relics

遗物适合在 BP 与奖励系统稳定后加入，因为它们通常是全局或队伍级 modifier。

第一批遗物应少而清楚：

- 全体物理单位攻击 +1。
- 全体魔法单位魔法攻击 +1。
- 每场战斗第一次破盾额外 +1 BP。
- 回合开始如果 BP 为 0，额外 +1 BP。
- 普通兵强化费用 -1 BP。

风险：

- 全局遗物很容易把数值推爆。
- 遗物叠加需要清楚的 modifier 管线。
- 遗物说明必须能被 hover 和日志解释。

## 13. Phase 8 - Weather / Battlefield

天气放最后，因为它会同时影响双方、影响节奏，还可能干扰 BP 测试结论。

可测试方向：

- 特定 Round 双方 AP +1。
- 特定 Round BP 获取效率提高，但仍受上限保护。
- 特定 Round 物理攻击 -1。
- 特定 Round 魔法伤害 +1。
- 特定 Round 护盾更容易破或更难破。

天气第一版应该是公开、可预测、持续时间短的全局规则，不要做隐藏随机惩罚。

## 14. 奖励池原则

奖励池未来统一服务所有成长内容。

第一版可以有：

- Common：普通兵、低费用强化、小属性奖励。
- Rare：英雄升级素材、指定兵种强化、较强普通兵。
- Epic：新英雄、高级英雄升级素材、强遗物。

奖励出现概率要服务两个目标：

- 让玩家经常有买得起的东西。
- 让强奖励偶尔出现，鼓励跳过和攒 BP。

保底机制可以存在，但不需要在 UI 中明文化。实现上可以记录玩家连续多少次没有看到英雄升级、新英雄、普通兵强化等关键奖励，用权重修正避免纯倒霉局。

## 15. UI 顺序

UI 接入也要按依赖走：

1. BP 常驻显示。
2. BP 获得/消耗反馈。
3. Dummy 三选一窗口。
4. 奖励 reset/skip。
5. 英雄/普通兵类型标识。
6. 锁定技能显示。
7. 开局选择界面。
8. 真实奖励卡面。
9. 升级选择界面。
10. 副官附着显示。
11. 遗物栏。
12. 天气/战场规则提示。

不要在一开始就把所有 UI 做成最终态。先保证信息读得懂，再逐步变漂亮。

## 16. 技术实现提醒

新增 BP 和奖励系统时要优先设计通用事件，而不是分散硬编码：

- `BpGained`
- `BpGainCapped`
- `BpSpent`
- `RewardWindowOpened`
- `RewardRerolled`
- `RewardSkipped`
- `RewardPurchased`
- `CardAcquired`
- `HeroUpgradeOffered`
- `HeroUpgraded`
- `SkillUnlocked`
- `SoldierStacked`
- `AdjutantAttached`

这些事件要能支撑：

- 本地 UI 刷新
- 在线同步
- 战况日志
- hover 解释
- 未来回放式演出
- 测试断言

## 17. 风险清单

### BP 来源太多

主动技能 +2、被动 +1、破盾 +1、造成伤害 +1 很容易让单回合收益爆炸，所以单回合上限 3 是硬保护。第一版不要因为“奖励玩家主动性”就放开上限。

### 被动刷 BP

被动技能如果在一次行动里多次触发，可能变成刷 BP 漏洞。需要定义“每个被动事件是否给 BP”和“同一角色同一被动每回合是否最多给一次 BP”。

### 奖励窗口打断节奏

每 2 Round 一次三选一可能频率偏高。先照计划做，因为对战 prototype 需要快速看到成长；如果实测打断太频繁，再调到每 3 Round 或只在奇数 Round。

### 新英雄和单位上限冲突

获取新英雄很兴奋，但队伍上限是 4。必须先设计替换/后备/拒绝领取，否则奖励会卡住。

### 英雄升级内容膨胀

8 个英雄、每个 2 个初级技能、2 个终极方向，很快变成 32 个以上内容点。第一版可以只完整做 1-2 个英雄，其余用占位结构验证系统。

### 副官过早加入会混乱

副官依赖普通兵强化和单位管理，不能提前到 BP 或开局选择之前。否则 UI 和规则都会卡在半成品状态。

## 18. 近期最小可执行路线

接下来建议严格按这个顺序走：

1. 实装 BP Core：状态、UI、获取、上限、每回合上限、日志。
2. 手动打一局，只看 BP 是否正常积累。
3. 实装 dummy 三选一奖励窗口：Round 1 首窗，之后每 2 Round，reset 一次，skip，购买只扣 BP。
4. 手动打一局，只看奖励窗口是否按时出现、是否正确扣 BP。
5. 接英雄/普通兵分类与技能锁定，不做完整成长内容。
6. 接开局选择：随机 3 英雄选 1，四初始兵种选 2。
7. 手动打一局，确认开局选择、锁定技能、普通兵认知负担都可接受。
8. 再把奖励接入真实内容：英雄升级、新英雄、普通兵获取、普通兵强化。
9. 等真实奖励稳定后，再做副官。
10. 最后做遗物和天气。

## 19. Phase 验收方式

每个 phase 都要有清楚验收，不只看代码能不能跑。

- 规则层：结果是否正确。
- UI：玩家是否能看懂。
- 日志：发生了什么是否能追溯。
- 预测：涉及伤害/属性时是否同步。
- 同步：双方状态是否独立。
- 数值：是否出现明显滚雪球或僵持。
- 文档：GDD 和 reference 是否更新。

## 20. 计划总结

新的路线不是“先把所有成长系统设计完”，而是先把 BP 作为局内成长货币跑通。只要 BP 能稳定积累、显示、扣除，三选一奖励窗口能稳定出现，后面的英雄升级、普通兵、招募、副官、遗物和天气才有共同入口。

因此第一目标非常具体：**先做一个没有真实奖励内容的 BP + dummy reward prototype**。它不好玩也没关系，但它必须可靠。可靠之后，再让内容进入系统。
