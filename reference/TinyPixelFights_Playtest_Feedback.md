# Tiny Pixel Fights — Playtest Feedback 整理与评估

> 记录日期：2026-06-21  
> 来源：与朋友进行的在线双人 playtest  
> 当前阶段：Prototype → 对照预期与实际体验 → 返回 Define／Ideate 迭代  
> 本文只整理问题、判断优先级与提出验证方向，不代表已经决定具体改法。

---

## 1. 本轮反馈应该怎样使用

Sections3 的流程不是“玩家提出什么就直接实现什么”，而是：

> **文档中的设计 → 设计者预期 → 实际游玩体验 → 定义问题 → 产生多个方案 → 再次 Prototype**

因此每条反馈需要拆成三层：

1. **观察／感受**：玩家实际看见、听见或感受到什么。这一层最应认真保留。
2. **原因判断**：玩家认为为什么会这样。这是重要线索，但可能不是唯一原因。
3. **建议方案**：玩家希望怎样修改。这是候选方案，不应在没有比较其他方案前直接成为规格。

同一条反馈还需要判断属于：

- **Bug**：规则或界面不能按预期工作。
- **Usability / Flow**：可以工作，但操作多余、难以理解或不适合目标设备。
- **Feedback / Legibility**：状态已经变化，但玩家不知道发生了什么、由谁造成或为何发生。
- **Balance / Pacing**：数值或行动经济使某些策略、角色或先后手明显占优。
- **Core Mechanics**：问题来自攻击、反击、AP 等核心规则本身，可能改变玩家的主要决策。

### 当前设计支柱

本轮判断继续以现有三个支柱为准：

1. **立刻行动**：回合短、操作直接，不保留没有选择价值的步骤。
2. **短而有分量的选择**：规则不需要重度复杂，但目标、顺序和资源投入应产生真实取舍。
3. **可应对的意外**：允许转折，但玩家应能理解发生了什么，并在之后调整。

目标体验现调整为：新玩家第一局约10分钟，熟练后约5分钟；快速、有来有回，偶尔出现令人惊讶但可以理解的局面。

## 2. 总体结论

本轮反馈不是零散的美术抱怨。它集中暴露了三个互相关联的问题：

### A. 战斗因果关系没有被清楚传达

攻击、技能、持续伤害、HP变化和死亡缺少完整的视觉／听觉反馈链。玩家无法稳定回答：

> 谁发动了什么 → 命中了谁 → 造成多少伤害 → 为什么死亡

这会直接破坏 Sections3 所说的互动循环：

> 玩家形成假设 → 采取行动 → 系统反馈 → 玩家修正理解

如果反馈缺失，即使底层技能规则正确，玩家也无法学习、评价或享受它们。因此这一问题会污染之后所有机制与平衡测试的结果。

### B. 战斗可能结束得早于角色机制被理解

玩家感觉 HP 低、集火强、先手强，角色可能在技能产生可感知价值以前离场。这不只意味着“数值不平衡”，也意味着当前 Prototype 没有给玩家足够的**理解 → 练习 → 掌握**时间。

V1.4 已将“提高全员HP”作为受控实验，但没有采用统一加点，而是把八人的基础体量按同一倍率×2并反推整数HP，精确保留原有体量比例。它仍可能与“熟练局约5分钟、不拖泥带水”的支柱冲突，因此F04尚不能视为解决；需要继续调查伤害、反击、先手、集火和角色技能显现速度。

### C. 一些低层操作正在阻碍节奏

无合法行动时仍需或无法手动结束回合、触摸界面无法收起详情等问题，并没有增加决策，只增加摩擦，直接违背“立刻行动”。Sections3 建议把熟练后不再构成选择的底层操作自动化，把注意力留给高层战术。

## 3. 逐条反馈分类与判断

优先级定义：

- **P0**：会造成软锁、无法理解关键结果，或令下一轮 playtest 数据失真；下次测试前应处理。
- **P1**：明显影响核心体验或公平性；应进入下一轮 Define／Prototype。
- **P2**：值得考虑，但需要更多样本或取决于平台范围。

### F01 — 参加方听不到 BGM

| 角度 | 判断 |
|---|---|
| 原始内容 | 在线对战的参加方听不到 BGM。 |
| 性质 | **Bug／平台技术问题／听觉反馈不一致** |
| 观察与建议的区分 | “参加方没有声音”是直接观察；目前没有足够证据判断是音频加载、浏览器自动播放限制、静音状态还是联网流程没有解锁音频。 |
| 对应 Sections3 | §29 Availability／Feedback：音效与音乐也是传达当前状态的渠道；§24 Beginner’s Mindset：不能假设玩家知道需要额外点击或解除浏览器静音。 |
| 对应支柱 | 间接支持“立刻行动”和整体氛围；不属于核心平衡，但会让两位玩家获得不一致体验。 |
| 是否纳入 | **是，P1。作为 Bug 调查，不作为机制改动。** |
| 理由 | 联机双方应获得一致的基础感官体验；如果浏览器要求用户手势，界面也应明确建立这一条件。 |
| 后续验证 | 分别记录房主／参加方、浏览器、是否点击过页面、AUDIO 状态、BGM文件请求结果和 AudioContext 状态。 |
| 处理状态 | **2026-06-22二次修复，待复测。** 除首次手势早于Audio对象创建的竞态外，又确认首次未缓存时“先静音预热、缓冲完成后异步解除静音”仍可能离开用户手势而被浏览器拦截。现在开始按钮会先登记BGM请求，再于同一次点击中直接发起有声播放；全局手势只重试已请求的BGM，不再抢先静音预热。 |

### F02 — AP 为 0 后仍需手动结束回合

| 角度 | 判断 |
|---|---|
| 原始内容 | AP 到 0 后已经没有可操作事项，再点“结束回合”是多余操作，应该自动结束。 |
| 性质 | **Usability／Flow／低层操作自动化建议** |
| 观察与建议的区分 | “没有选择却还要多点一次”是摩擦；“AP=0立刻自动结束”只是一个候选方案。 |
| 对应 Sections3 | §29 Action：减少操作步骤，输入速度匹配节奏；§30：底层操作可自动化，高层战术保留思考；§32：没有实际选择的按钮接近无意义决策。 |
| 对应支柱 | 直接支持“立刻行动”。 |
| 是否纳入 | **是，P1。应与 F08 合并定义为‘无合法行动时如何结束回合’，而不只判断 AP 是否为 0。** |
| 理由 | 真正的问题不是数字为 0，而是玩家是否还拥有任何合法且有意义的行动。 |
| 设计注意 | 自动结束必须等待当前攻击、技能、伤害和死亡演出结算完成；不能截断玩家理解结果的时间。 |
| 后续验证 | 列出所有合法行动：角色攻击、防御阵形、结束回合及未来可能新增的公共指令；建立统一的 `HasLegalAction` 判断。 |

### F03 — 攻击、伤害、技能与死亡缺少动画指引

| 角度 | 判断 |
|---|---|
| 原始内容 | 除拖拽箭头和护盾相关动画外，大多数操作没有动画。尤其在对方行动时，不知道谁攻击了谁、谁死亡、如何死亡；炎上可能让角色在回合开始时突然消失。希望至少有技能发动、通用掉血和死亡动画。 |
| 性质 | **Feedback / Legibility／战斗表现／初学者教学问题** |
| 观察与建议的区分 | “无法理解战斗因果”是核心问题；专属技能动画、掉血动画、卡片被切断只是适合进入 Ideate 的表现方案。 |
| 对应 Sections3 | §29 Feedback：必须传达选择后果；§31 Action / Feedback：攻击反馈应在不到1秒内快速明确；§24：游戏独有的技能和状态必须明确教学；§32：有后果但无反馈仍属于无意义决策问题。 |
| 对应支柱 | 同时支持三个支柱。没有清楚反馈，“意外”会变成莫名其妙；玩家也无法作出有依据的短决策。 |
| 是否纳入 | **必须，P0。下一次正式平衡测试前优先处理。** |
| 理由 | 当前表现不足会让玩家无法判断技能强弱、先手强弱和死亡原因，继续采集平衡反馈的可信度有限。 |
| 后续 Define 重点 | 建立统一事件表现顺序：来源提示 → 技能名／状态名 → 攻击轨迹 → 命中 → 盾／HP变化 → 死亡原因 → 退场。 |
| 处理状态 | **2026-06-22 已完成第一阶段实装，待联机 playtest 验证。** 八类 A 事件（物理、魔法、反击、技能、状态、治疗、护盾、死亡）已进入统一结算时间线；图标、既有攻击轨迹／受击效果、盾破碎和伤害数字共同播放。时间线来源于服务器结构化日志，因此非行动方也能看到同一因果顺序。后续仍可为各角色增加更具辨识度的专属 VFX。 |

#### F03 应拆分的反馈层级

| 层级 | 必须传达的信息 | 节奏要求 |
|---|---|---|
| 攻击宣言 | 谁攻击谁、物理还是魔法 | 立即，可与轨迹动画并行 |
| 技能发动 | 发动者、技能名、触发条件是否成立 | 短促醒目，不使用长演出 |
| 伤害结算 | 原始结果、盾吸收、实际掉血 | 直接显示在受击卡上 |
| 状态伤害 | 炎上等状态名称、伤害来源和目标 | 回合开始时先播原因，再扣血 |
| 死亡 | 谁死亡、直接死因、是否引发连锁技能 | 快速但不可直接消失 |
| 连锁效果 | 例如相邻伤害、守护、预见、衰弱 | 依结算顺序逐段提示，避免同时发生而看不懂 |

动画的目标是**压缩并澄清信息**，不是把短局游戏变成观看演出。专属动画可以使用共同骨架加职业化视觉，而不必为每个技能制作很长的独立过场。

### F04 — HP偏低，技能还没被理解战斗就结束

| 角度 | 判断 |
|---|---|
| 原始内容 | 几乎还没体会角色技能机制，战斗已经结束；例如公主可能第一回合就被集火击杀。 |
| 性质 | **Pacing／Balance／角色机制可见性／学习曲线** |
| 观察与建议的区分 | “角色机制没有足够时间显现”是重要观察；“HP太低”是可能原因之一。其他原因还可能是集火收益、反击结构、先手行动量、技能触发太慢或关键角色缺少保护窗口。 |
| 对应 Sections3 | §30 学习过程：理性学习 → 练习 → 自动执行；§34 Competence：理解 → 练习 → 掌握 → 新挑战；§22 Prototype：实际体验未达到预期，应返回 Define／Ideate。 |
| 对应支柱 | 与“快节奏”存在真实张力；也关系到“短而有分量的选择”是否来得及出现。 |
| 是否纳入 | **是，P1；必须调查，但暂不直接决定全员加 HP。** |
| 理由 | 如果角色经常在产生一次可感知技能价值前离场，专属技能系统就没有实现其设计目的。 |
| 后续验证 | 记录平均局长、回合数、每名角色存活回合、主动／被动技能触发次数、未触发即死亡比例、首个退场发生在第几个行动。 |
| 处理状态 | **2026-06-21 进入V1.4实验，待playtest。** 攻击与Cost不变，所有旧基础体量同比×2，再由体量反推HP；姫由5→12 HP，其余角色按各自旧体量获得不同增量。重点观察技能显现窗口与熟练局时长。 |

### F04a — 公主等关键被动角色容易被优先集火

| 角度 | 判断 |
|---|---|
| 性质 | **目标选择的明显答案／角色相性／集火收益** |
| 对应 Sections3 | §32 Obvious Decisions：若“先杀公主”长期全面优于其他目标，目标选择会失去意义；§33 Trade-offs／Yomi：集火应有收益，也应存在代价、反制或可预测的应对。 |
| 是否纳入 | **是，P1，作为独立于全局HP的问题观察。** |
| 判断标准 | 公主被优先攻击本身并不是错误；如果几乎每次都必须先杀、对方又没有合理保护或反制选择，才构成问题。 |
| 后续验证 | 统计各角色被选为第一攻击目标和第一退场角色的比例，并询问玩家为何选择该目标。 |
| 处理状态 | **2026-06-21 随V1.4进行缓解测试，尚未结案。** 姫仍保持全角色最低HP与最低基础体量，但从5提高到12 HP；“优先击杀姫”仍应有收益，只是不应再是低成本、无反制的默认答案。 |

### F05 — 先手过强，且先手如何决定不清楚

这一条包含两个不同问题，不能只用同一个改动处理。

#### F05a：先手优势

| 角度 | 判断 |
|---|---|
| 性质 | **Balance／Fairness／行动经济** |
| 玩家感受 | 先手方可以先集火并减少对方可用角色，使优势滚大。 |
| 对应 Sections3 | §33 Risk–Reward／Planning：后手应有调整计划，而不是在第一次行动前就失去关键资源；§34 Autonomy／Competence：玩家需要感觉结果来自决策而不是席位。 |
| 是否纳入 | **是，P1；先采样，再选择补偿方案。** |
| 后续验证 | 记录足够多局的先手胜率、首杀率、第一轮双方造成的净伤害与可行动角色差；区分新手和熟练玩家。 |

#### F05b：先手决定过程缺少说明与仪式

| 角度 | 判断 |
|---|---|
| 性质 | **Information／公平感／开局反馈** |
| 玩家感受 | 先手“莫名其妙就定好了”，即使数学上公平，主观上也缺少合理性。 |
| 对应 Sections3 | §29 Before／Availability：行动前应知道谁先行动；§24：不能要求玩家自行猜测规则；§32 Blind Decisions：席位结果不透明会被感受为盲目。 |
| 是否纳入 | **是，P1，但它只改善透明度，不能代替先手平衡。** |
| 候选验证方向 | 公开随机仪式、双方确认、轮换先手、由某种有代价的选择决定先手；进入 Ideate 后再比较。 |

### F06 — 等值反击降低进攻欲望

| 角度 | 判断 |
|---|---|
| 原始内容 | 进攻方会承受与防御方主动进攻时相同攻击力的反击，因此玩家不愿主动攻击。 |
| 性质 | **Core Mechanics／Risk–Reward失衡／消极策略风险** |
| 观察与建议的区分 | “玩家降低了进攻欲望”是需要重视的行为结果；“因为等值反击”是强假设，但还需结合护盾、击杀、Cost和目标选择验证。 |
| 对应 Sections3 | §32 Obvious Decisions：若不攻击／开盾成为长期唯一合理选择，会形成明显答案；§33 Risk–Reward：攻击风险必须与推进战局的收益匹配；§31 Short-term Loop：每回合应推动当前战术目标。 |
| 对应支柱 | 直接威胁“立刻行动”和“不拖泥带水”；但反击本身也可能支持有分量的目标选择，不能未经测试直接删除。 |
| 是否纳入 | **是，P1，属于核心机制级问题。** |
| 需要回答 | 玩家是不愿攻击所有目标，还是只回避高攻击目标？击杀免反击是否足以形成取舍？护盾是否进一步放大消极循环？低Cost多次攻击与高Cost重击是否受到不同影响？ |
| 后续验证 | 记录每回合攻击次数、开盾次数、主动提前结束次数、因反击死亡次数，以及玩家放弃攻击时的口述理由。 |
| 处理状态 | **V1.17待playtest。** 基础反击为`max（1，基础攻击-1）`；第一层防御阵型消耗2 AP获得盾2且不影响反击。若盾仍存在，第二层追加1 AP在当前盾值上+2，并为全员赋予一次反击-1的“有恃无恐”；若盾已被打空，再次防御会回到第一层。共享盾本身只吸收入伤。 |

### F08 — 有剩余 AP 但没有角色可以行动时，无法结束回合

| 角度 | 判断 |
|---|---|
| 原始内容 | 仍有 AP，但所有角色都不能行动时，结束回合不可用。 |
| 性质 | **Bug／Softlock／合法行动判断错误** |
| 对应 Sections3 | §29 Availability／Action：系统必须正确告知并允许当前可执行操作；§34 Autonomy：玩家不能被困在没有选择也无法推进的状态。 |
| 对应支柱 | 严重违反“立刻行动”。 |
| 是否纳入 | **必须，P0。下次 playtest 前修复。** |

## 5. 建议的处理顺序

### 第一阶段：恢复可测试性（P0）

1. [x] 修复“有 AP 但无法行动也无法结束”的软锁。
2. 建立最小但完整的战斗反馈链，至少能看懂攻击、技能／状态、掉血和死亡原因。

在这两项完成以前，角色强度与技能体验的反馈会持续受到 UI 误解和流程阻断污染。

### 第二阶段：清除明显摩擦（P1）

1. [x] 调查并修复在线参加方无 BGM。
2. 统一定义“是否还存在合法行动”，再 Prototype 自动结束回合。
3. 修复平板／触摸设备详情框的关闭逻辑。
4. 明示先手决定过程。

### 第三阶段：进行机制与数值实验（P1）

围绕三个问题分别建立小型 Prototype，不一次同时修改所有参数：

1. **角色机制来不及显现**：调查 TTK、HP、集火和技能触发窗口。
2. **先手优势**：调查首轮行动经济、首杀和是否需要后手补偿。
3. **反击压制进攻**：V1.5已小幅削弱基础反击；调查攻击收益、反击风险、护盾交互和消极策略。

这些问题相互影响。V1.4 HP扩张与V1.5反击削弱分别保留为独立版本记录，但下一轮实际测试会同时包含两者；分析时必须分别询问“角色是否活得更久”与“是否更愿意主动攻击”，不能把整体改善自动归因于其中一个改动。后手补偿暂不同时加入。

## 6. 下一轮 playtest 应记录的数据

目前反馈价值很高，但样本数、对局数和发生频率尚未记录。Sections3 提醒应寻找反复出现的模式，而不是只按单条评论修改。因此建议下一轮增加简短记录：

- 每名角色存活回合数

1. 哪一次死亡或伤害让你不知道原因？

3. 哪个角色还没来得及发挥就退场了？
4. 你是否知道谁先手，以及为什么？
5. 哪一步操作让你觉得多余或无法继续？

## 7. 下一轮需要验证的核心假设

| 编号 | 假设 | 成功信号 |
|---|---|---|
| H1 | 完整而短促的战斗反馈能让玩家理解技能与死亡因果，而不明显拖慢对局 | 玩家能复述主要死亡原因；熟练局仍接近约5分钟 |

## 8. 本轮暂不直接作出的决定

以下内容已经进入考虑，但尚未成为修改规格：

- V1.4 已采用一次按基础体量同比×2的HP扩张实验；在取得新局长、首杀与技能触发数据前，不继续追加HP或同时强化治疗／盾牌。
- V1.5已将基础反击统一调整为攻击-1、最低1；V1.7把盾牌的反击代价改为逐角色一次性Debuff“有恃无恐”。在取得攻击频率、Debuff消耗与盾牌交互数据前，不继续砍半或取消反击。
- 不直接决定后手获得额外 AP、护盾或其他补偿。
- 不把所有技能都做成长时间的独立演出。
- 不只用“AP是否为0”判断自动结束回合。

下一步应先修复 P0 问题，再为 HP／先手／反击分别进行受控的小范围 Ideate、Define 与 Prototype。
