# Tiny Pixel Fights

面向双人 playtest 的网页卡牌对战原型。后端使用 ASP.NET Core / C#，前端使用原生 HTML、CSS 与 JavaScript。

音频资产、事件映射与扩展方法见 [`reference/TinyPixelFights_Audio.md`](reference/TinyPixelFights_Audio.md)。
机制扩展、技能设计与 Cost/AP 可变性原则见 [`reference/TinyPixelFights_Extensibility_Principles.md`](reference/TinyPixelFights_Extensibility_Principles.md)。

本项目现在同时包含两种模式，并共用同一套战斗规则、角色、技能、界面、多语言资源和素材：

- 本地对战：两人在同一台电脑上轮流操作。
- 在线房间：两人分别使用自己的浏览器，通过房间邀请链接对战。

`DemoCardGame` 是后续开发的唯一主版本。原来的 `DemoCardGameOnlineTemp` 只作为整合前的备份参考，不再需要同步修改。

## 环境要求

- .NET SDK 9.0 或更高版本
- 现代桌面浏览器
- 推荐分辨率：1920×1080 以上；主要针对 2560×1440 优化

## 启动

在项目根目录执行：

```powershell
dotnet restore
dotnet run --urls http://localhost:5274
```

然后打开：

```text
http://localhost:5274
```

在封面选择“本地对战”或“在线对战”。语言也只在封面切换，选择会保存在当前浏览器中。进入任一对局后，需要点击画面中央的牌组才会开始发牌。

## 在线对战


### 临时发布到公网

先让服务监听本机端口：

```powershell
dotnet run --urls http://localhost:5274
```

Cloudflare Tunnel（无需改项目代码）：

```powershell
cloudflared tunnel --url http://localhost:5274
```

或使用 ngrok：

```powershell
ngrok http 5274
```

1. 主办方在封面选择“在线对战”，填写昵称并创建房间。
2. 将生成的邀请链接发给另一名玩家。
3. 对方打开链接、填写昵称并加入。
4. 两人到齐后，由主办方开始对局。

当前在线模式是轻量 playtest 版本：

- 一个服务器进程同时容纳一个两人房间。
- 房间和对局保存在内存中，重启服务器后清空。
- 玩家身份保存在各自浏览器的 localStorage 中。
- 两位玩家都始终从自己的视角观看战场；只有当前行动方能够操作。
- 只有主办方能够开始对局或发起再战。
- 每局只需任意一方触碰一次牌组，另一方会同步进入发牌演出。

如果要和不在同一局域网的网友玩，需要让对方能够访问主办方的服务地址，例如使用 HTTPS 隧道或部署到公网服务器。使用隧道时，请先通过公网地址打开游戏再创建房间，这样生成的邀请链接会自动带上正确的公网域名。

## 游戏概述

2 人回合制卡牌对战，高奇幻中世纪题材。8 名角色各唯一，随机平分 4v4。熟练玩家目标约 5 分钟一局。

### 核心规则

- 随机发牌保证双方至少各有一名 1 Cost 角色
- 每回合 **5 AP**，角色攻击消耗自身 Cost（1~3），AP 不保留
- 主动攻击与反击**同时结算**；基础反击 = `max(1, 基础攻击 - 1)`，之后经有恃无恐与预见等修正
- 伤害类型分为**物理**、**魔法**和**绝对**三种。绝对伤害绕过全部防御机制直接扣 HP
- 每名角色每回合最多行动一次；一方全部阵亡时对手获胜，可同时全灭导致平局

### 操作

- 拖拽可行动卡牌到敌方卡牌，或依次点击攻击者与目标
- 确认战斗预测后执行攻击
- 点击”回合结束”将行动权交给另一名玩家
- 防御阵形消耗 AP 展开共享盾
- `AUDIO` 可静音或恢复音乐

### 角色

| 角色 | Cost | ATK | HP | 类型 | 技能概要 |
|------|:---:|:---:|:---:|:---:|------|
| 農民 | 1 | 2 | 16 | 物理 | 首次攻击播种，下回合 ATK+2（隔回合循环） |
| 姫 | 1 | 1 | 12 | 物理 | 每己方回合全队 HP+1（上限+2），纯被动 |
| 占い師 | 1 | 1 | 14 | 魔法 | 全队预见减伤 + 赋予星读魔力（魔法主动攻击和炎上 +1） |
| ドルイド | 1 | 1 | 16 | 魔法 | 有伤100%/无伤50%发动：移除目标1个攻击强化Buff（不含野兽之怒），并令其下回合进攻−2 |
| 魔法使い | 2 | 4 | 16 | 魔法 | 攻击时 50% 附加炎上（延迟魔法伤害） |
| バーバリアン | 2 | 4 | 18 | 物理 | 造成 ≥3 伤害时波及相邻敌人 1 点 |
| 騎士 | 3 | 3 | 24 | 物理 | 每回合替队友挡一次主动物理伤害（代受 1 点） |
| モンスター | 3 | 3 | 22 | 物理 | 对非公主目标伤害为 0 时追击 3~4 绝对伤害；公主死亡期间永续 ATK+2（不可驱散） |

### 防御阵形（共享盾）

公共指令，不占角色行动次数。第一层消耗2 AP获得共有盾2；同回合若共有盾仍存在，追加1 AP会在当前盾值上+2。若共有盾已被打空，下一次防御指令会回到第一层，而不是强化第二层。每回合最多两次。

| 次数 | AP | 共享盾 | 有恃无恐 |
|:---:|:---:|:---:|:---:|
| 第一次 | 2 | **2** | 无 |
| 第二次（仍有盾时） | +1 | **当前盾值+2** | 反击 −1（每角色一次） |

- 盾池全队共享，吸收任意来源的正数伤害
- 有恃无恐逐角色独立：完成一次反击后仅该角色解除；未发动则在下个己方回合开始时消失
- 回合结束时盾和展开次数重置；若本回合内盾被打空，再次使用防御指令会重新展开第一层

### 伤害结算管线

```
1. 攻击方增减伤（衰弱 −2、占卜师魔法 +1）
2. 受击方预见概率 −1（物理25% / 魔法50%，优先级 10）
3. 共享盾吸收（优先级 20）
4. 骑士守护（优先级 20，仅主动物理 + 盾未完全吸收时）
5. 伤害 ≥ 0
```

绝对伤害跳过以上全部管线。

## 项目结构

```text
Api/                         共用 API DTO 与战斗预测
Domain/                      共用角色、状态、技能与战斗引擎
Services/GameSession.cs      本地对战 Session
Services/OnlineGameSession.cs 在线房间、席位与权限边界
assets/                      共用角色立绘与音频资源
reference/                   设计、平衡与技能文档
wwwroot/                     共用 HTML、CSS、JavaScript 与多语言资源
Program.cs                   两种模式的 API 路由与应用入口
```

角色数据位于 `Domain/CharacterDefinition.cs`，技能位于 `Domain/Skills.cs`，状态效果位于 `Domain/StatusEffects.cs`。本地和在线模式都调用同一个 `GameEngine`；模式差异只存在于 Session、玩家身份和操作权限层。

界面翻译集中在 `wwwroot/i18n.js`，角色、技能与状态通过稳定 ID 获取本地化文本。新增语言或修改 UI 文案只需维护这一份资源。

## 音频扩展

音频映射位于：

```text
wwwroot/config/audio.json
```

音频文件存放于 `assets/`。播放器支持 `bgm`、`sfx`、`ui` 独立音量总线，游戏事件与具体文件通过配置映射。

## 视觉风格

暗黑中世纪皇家沙盘——黑铁、旧金、象牙白、深红。日式二次元像素角色立绘（16/32-bit JRPG 风格），界面简洁锋利。UI 图标采用三层解耦：资产 ID → 语义绑定 → 实际 PNG，支持渐进式替换和失配占位符回退。

## 本地化

规则层与 API 只使用稳定 ID、枚举值和结构化消息，不保存日文或中文显示文本。语言资源位于：

```text
wwwroot/locales/ja.json
wwwroot/locales/zh.json
```

新增角色、技能、状态或战报时，需在兩份資源中加入相同 key。完整说明见 `reference/TinyPixelFights_Localization.md`。
